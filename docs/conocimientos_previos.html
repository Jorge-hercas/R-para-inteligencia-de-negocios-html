2  Conocimientos previos

Este libro estará enfocado en su mayoría en lenguaje R desde un nivel básico a intermedio, sin embargo, es importante complementarlo con ciertos conceptos que vamos a utilizar de forma recurrente a lo largo de los siguientes capítulos: Sintaxis de SQL en MySQL y SQL Server (consultas, manejo, modificación y creación de datos) y conocimientos estadísticos básicos.

Este conocimiento resulta esencial, ya que brinda una perspectiva integral sobre el manejo de información, facilitando la integración y manipulación de conjuntos de datos en distintos contextos.

Es cierto que, como lenguaje de programación, R nos permite manipular prácticamente en cualquier forma imaginable los datos con los que vayamos a trabajar, por lo que realmente no debemos preocuparnos demasiado de esta parte al trabajar con SQL, sin embargo, sería ingenuo pensar que en un entorno de ‘big data’ actual no nos encontraremos con bases de datos relacionales (incluir pie de página) y no relacionales, razón por la que me parece necesario el tener clara una base de conocimientos en SQL que, en capítulos posteriores, podremos integrar a nuestro entorno de trabajo, combinándola con otros lenguajes de programación.

2.1 ¿Qué es SQL?

En forma resumida, SQL (por las siglas Structured Query Language) es un lenguaje para consulta y manejo de bases de datos relacionales, es decir, aquellas en las que existen tablas con relaciones entre si (a partir de llaves o ID’s). Los usos de este lenguaje pueden ir desde consultar una cierta cantidad de datos hasta crear tablas, columnas, resumenes de ellas e incluso funciones más complejas (Groff, Weinberg, and Oppel 2002).

El objetivo de esta sección será brindarte de un conocimiento básico/intermedio sobre consulta y manipulación de información, lo que te permitirá combinarlo con R más adelante.

Ahora, SQL es en si el lenguaje general, pero existen diferentes sistemas de administración de bases de datos con pequeñas variaciones en su sintaxis o funciones diferentes, como PostgreSQL, SQL Server, SQLite y MySQL, siendo este último en el que nos vamos a enfocar para esta sección.

2.2 SQL ¿por qué debería usarlo?

Las hojas de cálculo son geniales, ya que nos permiten manipular información de forma rápida y, casi siempre, con bastante precisión. Estos archivos nos permitirán el trabajar con análisis de datos de manera eficiente, pero solo hasta un cierto punto. En proyectos personales y/o escolares generalmente una hoja de cálculo será más que suficiente para almancenar la información requerida para nuestro análisis, pero en ambientes empresariales, donde los datos se actualizan por hora, minutos o incluso segundos, donde además el volumen de información considerablemente más alto de lo que un archivo de excel es capaz de soportar, es cuando el tener conocimientos de un lenguaje de gestión de bases de datos, como lo es SQL, se vuelve algo crucial en el día a día.

SQL no es un lenguaje de programación como tal (aunque en el sentido estricto tiene funciones que lo podrían hacer entrar a esta categoría), pero si nos permitirá manipular bases de datos de prácticamente cualquier forma imaginable. Una vez que comprendas y domines los conceptos básicos que analizaremos en esta sección previa, serás capaz de extraer información de bases de datos relacionales de una forma bastante eficiente y sencilla, ya que si, la mayoría de las consultas que harás en SQL a lo largo de tu vida profesional no serán muy complicadas (y serán autoexplicativas). El verdadero reto surgirá cuando lleves esa información a R y decidas explotar tus datos mediante análisis.

2.3 Descargando un gestor

Antes de comenzar a ver código, comencemos por la parte medular: instalar MySQL en tu computadora. La elección de este gestor radica principalmente en la posibilidad de utilizar su versión gratuita, la cual usaremos para (casi todos) nuestros proyectos a lo largo de este libro.

Para instalar esta versión, simplemente debemos hacernos una cuenta en la página de Oracle (dueño del gestor MySQL actual) y seleccionar la versión community compatible para nuestro ordenador, de su sitio de descargas. Además de ello, también es recomendable instalar el programa Workbench, el cuál nos permitirá trabajar con MySQL y su código de una forma mucho más amigable.

Para poder utilizar un servidor local será necesario activar la instancia correspondiente en la configuración de nuestro equipo, aunque durante la instalación podemos establecer por default que siempre permanezca activa, lo que consumirá recursos pero ahorrará algo de tiempo.

Durante tu vida profesional (casi) siempre harás uso de instancias de bases de datos alojadas en la nube (AWS, AZURE, etc), donde estarán todos los registros de la empresa, por lo que lógicamente no tenemos acceso a servidores en la nube ahora mismo, razón por la que vamos a utilizar nuestra propia computadora para crear bases de datos locales. Más adelante veremos como configurar instancias en la nube, sin embargo, esos servicios son costosos, por lo que por ahora nos limitaremos a un uso local que te permita entender la lógica detrás de este lenguaje.

2.4 De lo general a lo particular: Desde una base de datos hasta una observación.

Ahora que sabes qué es y para qué funciona este lenguaje, veamos de una forma más gráfica su estructura. Como sabemos, una base de datos no es más que un conjunto de, valga la redundancia, datos almacenados (casi siempre) de forma estructurada.

Una base de datos no es más que una recopilación estructurada de información (datos), la cual puede o no estar estructurada en un formato de tablas relacionadas entre si (tema que veremos con mayor detalle más adelante). Crear una base de datos en SQL es tan sencillo como utilizar el siguiente código:

CREATE DATABASE nombre_base_ejemplo;

Cuando nosotros accedemos a una base de datos como, por ejemplo, la BD de la empresa en la que estes trabajando, encontraremos que tenemos diferentes esquemas (conjuntos de tablas, procedimientos, etc) que tendrán diferentes objetivos. La idea de separar la base de datos en esquemas es, principalmente, para tener un orden en nuestros datos tanto en su estructura como en la seguridad y acceso de ellos. Para crear un esquema podemos hacer uso del siguiente código:

CREATE SCHEMA mi_esquema;
Nota

A pesar de que los esquemas separan en cierta medida los datos, los ambientes de prueba suelen estar almacenados en una base de datos adicional a la del ambiente de producción, esto debido a que las tablas pertenecientes a diferentes esquemas aun pueden relacionarse, por lo que no es recomendable crear esquemas de prueba, ya que existe el riesgo de afectar información importante.

Como es lógico, el siguiente nivel en nuestra base de datos serán las tablas. Una tabla es una estructura sencilla que nos permitirá almacenar datos mediante columnas y filas, siendo estos clasificados por tipo de dato (ej. varchar, decimal, date, etc). Al ser una base de datos relacional, podemos establecer relaciones entre tablas a partir de llaves primarias y llaves foráneas, es decir, los ID’s de una tabla. Cuando hablamos de llaves primarias nos referimos al ID de cada observación perteneciente a una tabla, mientras que cuando hablamos de llaves foráneas nos referimos a un ID en una tabla que hace referencia al ID primario de otra, por ejemplo: Supongamos que tenemos una tabla llamada usuarios con nombres de usuario que incluye una columna llamada tipo_usuario_id con ID’s 1 o 2, referentes a otra tabla llamada tipo_usuarios en la que el ID 1 equivale a usuario maestro y el ID 2 a usuario general, en tal caso la columna tipo_usuario_id de la primer tabla será una llave foránea, ya que hace referencia a los datos de otra tabla, mientras que la columna id de la tabla tipo_usuarios será la llave primaria. Gráficamente se puede ver esta relación en el diagrama Figure 8.13.

Para crear una tabla debemos tener en cuenta los tipos de columnas que vamos a crear, así como las relaciones que van a existir en dicha tabla respecto a otras ya existentes (o no). Los tipos de columnas disponibles más comunes son:

  • INT: Número entero.
  • NUMERIC o DECIMAL: Números con decimales de longitud exacta. Se requiere establecer la cantidad de dígitos y la cantidad de dígitos posteriores al punto decimal.
  • DOUBLE: Números muy grandes o muy pequeños con mayor precisión.
  • CHAR: Cadena de caracteres con longitud establecida.
  • VARCHAR: Cadena de caracteres con longitud variable (aunque con un máximo de caracteres).
  • DATE: Fechas en formato aaaa-mm-dd.
  • TIME: Tiempos en formato hh-mm-ss.
  • DATETIME: Fechas y horas del día en formato aaaa-mm-dd-hh-mm-ss.
  • BOOLEAN: Valores binarios como TRUE o FALSE.
  • JSON: Formato de información jerárquica.

Estos únicamente representan a grandes rasgos los tipos de datos más comunes con los que nos podremos topar en una tabla (los cuales vamos a analizar con detalle en capítulos posteriores), aunque existen muchos otros para usos más específicos, como NCHAR, NVARCHAR, BIT, BIGINIT, etc (Converse, Park, and Morgan 2004).

Ahora bien, al crear una tabla en SQL tendremos que establecer el tipo de columnas a introducir de manera directa (ya que estas no pueden tener más de un tipo de dato), por ejemplo: Supongamos una base de datos que tenga un esquema llamado usuarios_data, en el cual podemos crear una tabla llamada usuarios, tal que:

CREATE SCHEMA usuarios_data;

CREATE TABLE usuarios_data.departamentos(
  ID INT NOT NULL,
  Departamento_nombre VARCHAR(10),
  PRIMARY KEY(ID)
);

CREATE TABLE usuarios_data.usuarios(
  ID INT NOT NULL,
  Nombre_usuario VARCHAR(30),
  Apellido VARCHAR(30),
  Dep_id INT,
  Fecha_creacion DATE,
  PRIMARY KEY(ID),
  FOREIGN KEY(Dep_id) REFERENCES usuarios_data.departamentos(ID)
);

Como puedes observar, la lógica para crear tablas en SQL es bastante sencilla y autodescriptiva. En este caso tenemos una llave foránea en la columna Dep_id por ID de puesto que hace referencia a una tabla índice que contiene los nombres de cada departamento con su respectivo ID.

La lógica establecida anteriormente es referente a MySQL, pero en casos como SQL Server estableceremos a las llaves primarias en primera instancia, tal que:

CREATE TABLE usuarios_data.departamentos (
  ID INT NOT NULL PRIMARY KEY,
  Departamento_nombre VARCHAR(10)
);

Lo que hará más compacto el código.

Obviamente, el siguiente paso lógico al ya tener las tablas que vamos a utilizar es añadir las observaciones (filas), trabajo que resulta mucho más sencillo mientras sigamos las reglas que nosotros mismos establecimos. La tabla usuarios tiene un pequeño problema, la columna Dep_id está ligada a la tabla departamentos, pero ahora mismo esa tabla se encuentra vacía, por lo que sin importar lo que nosotros ingresemos al querer introducir datos a la tabla usuarios se nos devolverá un error (excepto si mantenemos un valor NULL), ya que este valor no existirá en la tabla de referencia a la que se encuentra conectado, por lo tanto, vamos a comenzar por añadir información a la tabla departamentos mediante la orden INSERT INTO, por ejemplo:

INSERT INTO usuarios_data.departamentos (ID, Departamento_nombre) 
       VALUES (1, "Contabilidad");

Donde el primer argumento entre paréntesis indicará las columnas que vamos a afectar, mientras que el segundo argumento indicará los valores a introducir. En este caso insertamos el ID de la columna de manera manual, sin embargo, podemos automatizar esta columna mediante la orden AUTO_INCREMENT al definir nuestra tabla, garantizando la integridad de dicha columa, tal que:

CREATE TABLE usuarios_data.departamentos (
  ID INT AUTO_INCREMENT PRIMARY KEY,
  Departamento_nombre VARCHAR(10)
);

Ya que tenemos información en la tabla departamentos entonces podemos comenzar a llenar la otra tabla mediante una sentencia similar, tal que:

INSERT INTO usuarios_data.usuarios (ID, Nombre_usuario, 
                                    Apellido, Dep_id, 
                                    Fecha_creacion) 
       VALUES (1, "Nombre ejemplo", "Apellido ejemplo", 
               1,CURDATE());

Donde la función CURDATE() nos devolverá la fecha actual de manera automática.

Ahora podemos decir que comprendemos a nivel general la estructura de una base de datos, aunque claro, existen más componentes como las vistas, tablas temporales, procedimientos almacenados, cursores y un gran etc que por el momento no vamos a abordar, ya que este libro está enfocado principalmente en lenguaje R, pero por el momento podemos avanzar a una parte que probablemente te será de mayor utilizad: realización de consultas de datos.

2.5 Consultas de tablas.

Ya que conocemos como funcionan las bases de datos y sus tablas, podemos comenzar a extraer información de ellas de forma muy simple. Para entender mejor esta parte vamos a suponer que las tablas que creamos anteriormente ya cuentan con más información, tal que:

Tabla de usuarios
ID Nombre_usuario Apellido Dep_id Fecha_creacion
1 Luis Robles 1 2024-01-01
2 Chris Suarez 1 2024-01-01
3 Mario Lombardo 2 2024-01-01
Tabla de departamentos
ID Departamento_nombre
1 Contabilidad
2 IT

Problemente ya sabes que mediante una consulta SELECT * FROM usuarios_data.usuarios obtendremos toda la información proveniente de la tabla, lo que simplifica bastante las cosas, pero también implica que, muy probablemente, obtendremos información que no necesitamos, lo que de igual forma va a consumir recursos y alentará la extracción de la información. Puede que en esta situación no se note ese problema, pero en tablas donde tenemos cientos de miles de registros y una gran cantidad de columnas es donde realmente vamos a querer ser muy específicos respecto a la información que vayamos a consultar. Supongamos que únicamente queremos el nombre de usuario y el ID de su puesto, en ese caso simplemente escribimos el nombre de las columnas, tal que:

SELECT Nombre_usuario,Dep_id
FROM usuarios_data.usuarios

Una consulta bastante sencilla, pero poco práctica cuando queremos hacer análisis de esta información. Sabemos que el ID se refiere a un puesto en específico, pero no sabemos cuál al ver la tabla a simple vista. Si qusiéramos ser más claros con ello podríamos hacer uso de un condicional CASE WHEN para reemplazar los valores por nombres, tal que:

SELECT Nombre_usuario, 
       CASE WHEN Dep_id = 1 THEN "Contabilidad"
            WHEN Dep_id = 2 THEN "IT"
       END as Puesto_nombre
FROM usuarios_data.usuarios

En este caso estamos indicando el valor a introducir cuando la columna Dep_id es igual a un cierto ID, sin embargo, el uso de estos condicionales solo es recomendable en situaciones muy generales, donde se requiera de pocos valores únicos. ¿Qué podríamos hacer si existieran más de 20 o 30 puestos en la empresa? Crear un condicional para cada caso sería una mala idea. En este punto vamos a introducir un concepto interesante: Uniones de tablas, conocidas más comunmente como JOINS.

2.5.1 JOINS

Este tipo de operaciones en SQL son una forma en la que podemos unir valores de tablas a partir de una (o más) coincidencia(s), algo parecido a lo que hace una función `buscarV` o `buscarX` en excel. Por supuesto son bastante sencillas de usar, pero debemos entender su funcionamiento para poder obtener el resultado deseado.

Los JOINS no son más que uniones (e intersecciones) basadas en teoría de conjuntos. Un conjunto es una colección de elementos con características definidas pertenecientes a un mismo espacio, por lo que lógicamente podemos definir a una tabla como un conjunto, el cual va a respetar las propiedades de los conjuntos (véase el apéndice).

Al unir dos conjuntos (o tablas) mediante un JOIN, el tipo de resultado deseado será lo que determinará el tipo de JOIN que vayamos a utilizar. Supongamos que tenemos una tabla A en la que queremos añadir información coincidente de una tabla B, es decir, combinar las filas de ambas tablas a partir de una coincidente entre ambas, en tal caso los diferentes tipos de uniones nos devolverán los siguientes resultados:

  • LEFT JOIN: Todas las filas de la tabla A y únicamente las filas coincidentes de la tabla B.
  • RIGHT JOIN: Todas las filas de la tabla B y únicamente las filas coincidentes de la tabla A (el inverso de LEFT JOIN).
  • FULL JOIN: Todas las filas de las tablas A y B, haya o no coincidencia.
  • INNER JOIN: Únicamente las filas de ambas tablas A y B donde haya coincidencia.

Aunque por supuesto, en estos casos una explicación no es lo suficientemente explicativa a mi consideración. Vamos a ilustrar las uniones de una forma más clara mediante diagramas de Venn (véanse las figuras de la siguiente sección), los cuales nos muestran, mediante conjuntos representados por círculos (nuestas tablas) la forma de unión de dicha información, donde el área de color rojo representa las filas devueltas.

El uso de un tipo de JOIN estará en función de lo que necesitamos. En la subsección anterior mencionamos el uso de tablas de referencia que se conectarán a nuestas tablas principales; En la mayoría de los casos partiremos de nuestra tabla principal y utilizaremos un LEFT JOIN para conectar información de tablas de referencia sin perder información de la tabla principal. Al ser RIGHT JOIN la versión invertida, serán raros los casos en los que hagamos uso de esta operación (a menos de que partamos de nuestra tabla de referencia). Lógicamente, al ser INNER JOIN una forma en la que devolvemos únicamente filas en las que se encontró una coincidencia entre ambas tablas, podemos hacer uso de esta operación para filtrar nuestra tabla principal en ciertos casos.

La sintaxis de uso de un JOIN en SQL es bastante sencilla, simplemente debemos declarar las columnas mediante las cuales haremos la unión de tablas. Regresemos al caso que creamos al principio con dos tablas: usuarios y departamentos. Si queremos unir ambas tablas para que la columna Departamento_nombre aparezca en la tabla de usuarios, considerando como intersección a Dep_id de esta última y ID de la tabla departamentos, entonces utilizaríamos la siguiente consulta:

SELECT * FROM usuarios
LEFT JOIN departamentos 
      on usuarios.Dep_id = departamentos.ID

Claro que esta consulta devolvería todas las columnas de ambas tablas, lo que puede no siempre puede ser la situación más conveniente cuando tenemos demasiada información, por lo que podemos limitar las columnas seleccionadas de la siguiente manera:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
LEFT JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID

Nótese que establecí un identificador más corto en cada tabla (usr y dep), esto para simplificar la consulta realizada. Como puedes observar, cuando extraemos información mediante JOINS es necesario utilizar como prefijo de cada columna el nombre (o alias) de la tabla seleccionada, para que SQL pueda interpretar de donde viene la información consultada.

El uso de los JOIN’s no está limitado a las columnas de ID’s, podemos utilizarlo en todas aquellas columnas donde podamos encontrar una relación, sin embargo, considera que al hacer uso de combinaciones por coincidencias, si tenemos situaciones en las que existe más de un valor que coincida con el de nuestra tabla principal, este se va a ligar a la misma observación creando un duplicado de la fila, por lo que debemos ser cuidadosos al realizar estas operaciones.

2.5.1.1 Algunos ejemplos.

Ahora que sabemos cómo unir información de diferentes tablas, podemos pasar a situaciones más prácticas donde pondremos a prueba el uso de los JOIN’s. Regresemos a nuestras tablas anteriores, suponiendo ahora que tenemos las siguientes observaciones:

Tabla usuarios
ID Nombre_usuario Apellido Dep_id Fecha_creacion
1 Victor Ruiz 1 2024-03-03
2 Yuriko Mendoza 2 2024-03-04
3 Darlene Archuleta 1 2024-03-05
4 Melissa Bravo 3 2024-03-06
5 Shayna Gonzalez 1 2024-03-07
Tabla departamentos
ID Departamento_nombre
1 Contabilidad
2 IT
4 Ventas

Por lo tanto, si utilizamos la consulta explicada anteriormente, es decir:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
LEFT JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID

Tendremos el siguiente resultado:

Resultado de un LEFT JOIN

2 Yuriko Mendoza 2024-03-04 IT
3 Darlene Archuleta 2024-03-05 Contabilidad
4 Melissa Bravo 2024-03-06 NULL
5 Shayna Gonzalez 2024-03-07 Contabilidad

Por supuesto que, al no tener un ID de departamento igual a 3, simplemente se nos devolverá un espacio vacío. Si utilizamos la contraparte RIGHT JOIN entonces estaremos viendo como tabla principal a la tabla de referencia departamentos, por lo que no se perderán observaciones de ese lado pero si del lado de la tabla usuarios, tal que:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
RIGHT JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID

Resultando en:

Resultado de un RIGHT JOIN
ID Nombre_usuario Apellido Fecha_creacion Departamento_nombre
1 Victor Ruiz 2024-03-03 Contabilidad
2 Yuriko Mendoza 2024-03-04 IT
3 Darlene Archuleta 2024-03-05 Contabilidad
5 Shayna Gonzalez 2024-03-07 Contabilidad
NULL NULL NULL NULL Ventas

Esto aún nos deja con espacios vacíos en ambos casos, por lo que si quieremos evitar estos huecos, podemos hacer uso de un INNER JOIN y asegurarnos de únicamente retornar filas donde haya una coincidencia, es decir:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID

Resultando en:

Resultado de un INNER JOIN

1 Victor Ruiz 2024-03-03 Contabilidad
2 Yuriko Mendoza 2024-03-04 IT
3 Darlene Archuleta 2024-03-05 Contabilidad
5 Shayna Gonzalez 2024-03-07 Contabilidad

Como pudiste notar en las tablas por separado, el ID de departamento 3 no existe. En ciertas ocasiones, cuando trabajamos con limpieza de datos, a veces buscaremos específicamente casos en los que exista una falta de información, esto para después hacer otros arreglos y correcciones de datos; En estos casos específicos podemos hacer uso de un ANTI JOIN, operación que nos devolverá aquellas observaciones donde no exista una coincidencia, tal que:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
ANTI JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID

Resultando en:

Resultado de un ANTI JOIN
ID Nombre_usuario Apellido Fecha_creacion
4 Melissa Bravo 2024-03-06

2.5.2 Filtros WHERE

Claro, una combinación de tablas, incluso mediante un INNER JOIN, no siempre es suficiente cuando queremos ser más específicos respecto a los resultados de nuestra consulta.

La instrucción WHERE será de los elementos que más utilizaremos al trabajar con consultas y modificación de tablas, ya que nos permitirá filtrar información de forma sumamente específica mediante el uso de otros operadores. Pensemos en nuestra penúltima consulta en la que utilizamos un INNER JOIN para filtrar los datos a aquellos en los que existiera una coincidencia, ¿y si solo quisiéramos ver las filas en las que el departamento es igual a IT? En este caso simplemente tendríamos que filtrar mediante una instrucción WHERE, tal que:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE dep.Departamento_nombre = 'IT'

Lo que nos devolvería el siguiente resultado:

Resultado de un INNER JOIN
ID Nombre_usuario Apellido Fecha_creacion Departamento_nombre
2 Yuriko Mendoza 2024-03-04 IT

Para segmentar la información de forma directa basta con que especifiquemos una igualdad en el filtro, aunque existen otros operadores de los que podemos hacer uso, por ejemplo: Supongamos que queremos devolver las filas creadas en los últimos 2 días o posterior a ello, en tal caso podemos hacer uso del operador >= para indicar una equivalencia entre la columna de fechas y la fecha actual en la que se buscarán valores mayores o iguales, es decir:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE usr.Fecha_creacion >= NOW() - INTERVAL 2 DAY

Como puedes notar, en este caso hicimos uso de la función NOW() para obtener la fecha actual, a la que le restamos dos días mediante la instrucción INTERVAL 2 DAY, lo que de esta forma nos devolverá siempre aquellas observaciones creadas en los últimos dos días, sin importar la fecha actual. En la mayoría de los casos el uso de este tipo de funciones nos simplificará el trabajo en la extracción y filtrado de datos, ya que no tendremos que preocuparnos por hacer modificaciones a una consulta cuando decidamos extraer información en un futuro.

De forma simplificada, los operadores disponibles a utilizar (más los ya vistos hasta ahora) para filtrar nuestras consultas son los siguientes:

Operadores disponibles en SQL
Operador Significado
= Igualdad

<= Menor o igual que
> Mayor que

El punto de tener diferentes operadores es, simplemente, para poder cubrir todas las necesidades que podamos necesitar en cuanto a la segmentación de datos, por ejemplo: Cuando queremos filtrar una tabla con más de un elemento a buscar no podemos hacer uso de una igualdad, en ese caso tendríamos que usar el operador IN de la siguiente manera:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE dep.Departamento_nombre IN ('IT', 'Contabilidad')

En un caso opuesto donde quisiéramos obtener todos los resultados diferentes a uno o más elementos, simplemente debemos añadir un NOT previo al filtro, tal que:

SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE dep.Departamento_nombre NOT IN ('IT', 'Contabilidad')

Claro que la búsqueda (u omisión) de estos datos en ciertas ocasiones puede ser más ambigua, tratando de encontrar ciertas palabras clave o terminaciones comunes de frases, por lo que podemos hacer uso de regulares para ello junto al operador LIKE, tema que analizaremos con detalle en capítulos posteriores, al ser demasiado complejo en este punto del libro.

Veamos una situación algo más interesante. Supongamos que queremos devolver únicamente las filas donde hubo coincidencia, pero esto en una consulta que utilice un LEFT JOIN, en tal caso podemos explícitamente dar dicha orden en un filtro con el uso de not null, aunque hay un “pero” en este caso, no importa si eliminamos los valores NULL de una o ambas tablas, seguiríamos teniendo estos valores después de hacer la unión, ya que estos se generan al no encontrar una coincidencia entre ambas fuentes. En este caso específico podemos eliminar dichos NULL una vez realizada la unión de ambas tablas mediante el uso de una subconsulta o subquery. Una subconsulta no es más que una operación realizada a otra consulta ya establecida, encapsulando a la primera mediante alias, es decir:

SELECT * FROM (
  SELECT usr.ID, usr.Nombre_usuario, 
       usr.Apellido, usr.Fecha_creacion, 
       dep.Departamento_nombre
FROM usuarios usr
LEFT JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
) AS subquery
  WHERE Departamento_nombre IS NOT NULL

Resultando en:

ID Nombre_usuario Apellido Fecha_creacion Departamento_nombre
1 Victor Ruiz 2024-03-03 Contabilidad
2 Yuriko Mendoza 2024-03-04 IT
3 Darlene Archuleta 2024-03-05 Contabilidad
5 Shayna Gonzalez 2024-03-07 Contabilidad

En este caso, lo único que hicimos fue encerrar entre paréntesis nuesta consulta original y asignarle un alias, haciendo que la consulta secundaria la considere como una única tabla a la cual si podemos filtrarle los valores NULL después de la unión de tablas.

El uso de WHERE no solo se limita a consultas de información. Cuando queremos actualizar un registro específico en una tabla utilizaremos esta instrucción para indicar a que fila (o filas) se va a aplicar dicho cambio, por ejemplo: Si quisiéramos actualizar el departamento de procedencia del usuario con ID igual a 2 por el de contabilidad, simplemente debemos utilizar la instrucción UPDATE de la siguiente manera:

UPDATE usuarios
SET Dep_id = 1
WHERE ID = 2

Afectando únicamente a la fila que cumpla con dicha condición, estableciendo el nuevo valor de Dep_id, por lo que ahora estaría coincidiendo con el departamento de contabilidad.

Cuidado

Cuando trabajes con actualizaciones de registros siempre ten presente a qué fila(s) quieres afectar mediante un filtro WHERE, ya que en caso de no incluirlo estaríamos actualizando los valores de toda la columna por el valor que vayamos a establecer en la instrucción. No tomes a la ligera estas instrucciones, ya que una omisión de este estilo podría arruinar los registros de una tabla.

Como consejo personal, te recomiendo que en estos casos manejes las actualizaciones de datos mediante transacciones, ya que esto te permitirá revertir cambios en caso de cualquier error mediante un comando ROLLBACK, por ejemplo:

# Iniciamos una transacción
START TRANSACTION;

# Actualización erronea
UPDATE usuarios
SET Dep_id = 1
WHERE ID = 2;

# Revertimos los cambios realizados
ROLLBACK;

2.5.3 GROUP BY

Los filtros solo son el primer paso cuando trabajamos con datos, el siguiente nivel será resumir la información para que sea entendible para otros usuarios. Generalmente resumiremos información a partir de columnas que sean más generales mediante agrupaciones de sus elementos GROUP BY que veremos a continuación. Un GROUP BY simplemente agrupa (valga la redundancia) los datos de una tabla a partir de una columna donde existan elementos en común; A partir de la agrupación realizada podemos obtener otros cálculos, como promedios, sumas, la última observación por grupo, etc. A este tipo de cálculos se les conoce como funciones de agregación, ya que únicamente devuelven un solo valor para todo el conjunto de datos (o para cada elemento de una agrupación).

Imagina una situación en la que tenemos una tabla bastante grande donde una tienda registra las ventas realizadas a cada cliente, incluyendo cada producto comprado por separado, tal que:

Ejemplo de tabla de ventas
Cliente Producto Precio
Cliente_3 Pasta 26
Cliente_3 Sopa 26
Cliente_3 Fruta 21
Cliente_1 Pasta 25
Cliente_5 Pasta 25
Cliente_1 Agua 22

Si quisiéramos obtener el total de dinero gastado por cada cliente, podemos realizar una agrupación en una consulta sencilla de la siguiente manera:

SELECT Cliente, SUM(Precio) as Total
FROM ventas
GROUP BY Cliente

Resultando en:

Resultado de un agrupamiento
Cliente Total
Cliente_1 47
Cliente_3 73
Cliente_5 25

Regresemos a las tablas creadas anteriormente (usuarios y departamentos), supongamos que queremos averiguar el número de trabajadores registrados en el departamento de contabilidad, para ello simplemente debemos hacer una consulta que combine ambas tablas y filtre los departamentos como lo hicimos en la sección anterior, para después agrupar por esta última columna, es decir:

SELECT dep.Departamento_nombre, COUNT(*) conteo
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE dep.Departamento_nombre = 'Contabilidad'
GROUP BY dep.Departamento_nombre

Lo que nos devolvería lo siguiente:

Conteo de usuarios
Departamento_nombre conteo
Contabilidad 3

En este caso sabemos que el resultado es correcto, ya que únicamente tenemos un usuario con un ID único, pero ¿y si no tuvieramos la seguridad de ello? En caso de tener una tabla con valores duplicados podemos hacer uso de una orden DISTINCT para que se haga un conteo por valor único, por ejemplo:

SELECT dep.Departamento_nombre, COUNT(DISTINCT usr.ID)
FROM usuarios usr
INNER JOIN departamentos dep
      on usuarios.Dep_id = departamentos.ID
WHERE dep.Departamento_nombre = 'Contabilidad'
GROUP BY dep.Departamento_nombre

Asegurándonos de obtener un resultado exacto en cuanto al número de usuarios registrados para este ejemplo.

2.5.3.1 Otro tipo de agrupaciones

El uso de GROUP BY no solo se limita a resumir información de una tabla, también podemos utilizarlo para obtener datos posicionados en un cierto lugar. En ciertas ocasiones vamos a querer buscar un valor (o valores) que tenga algo en particular, momentos en los que podemos apoyarnos de agrupaciones para poder obtener esta información. Regresemos al ejemplo de la tabla de ventas con productos comprados por clientes, supongamos que queremos devolver la compra más reciente hecha por un cliente (suponiendo ahora que la tabla cuenta con una columna denominada Fecha_compra), por lo que podemos utilizar la siguiente consulta:

SELECT Cliente, MAX(Fecha_compra) as Compra_reciente
FROM ventas
GROUP BY Cliente

Resultando en:

Compra con valor máximo
Cliente Compra_reciente
Cliente_1 2023-03-03
Cliente_3 2023-05-11
Cliente_5 2023-04-19

Este sigue siendo un uso sencillo al solo tener que establecer una función (MAX()) para obtener los datos buscados por fecha, pero ¿y si quisiéramos extraer la primera observación de cada compra por cliente sin importar la fecha? Cuando queremos extraer datos por posición independientemente de que estos cumplan una condición de valor (más reciente, valor máximo/mínimo, etc), podemos hacer uso de un filtro por número de fila, mediante la función ROW_NUMBER(), la cuál nos devolverá dicho número, con el que podremos agrupar a partir de otra columna mediante la instrucción OVER (PARTITION BY "columa_nombre"). En este caso, si agrupamos por el cliente entonces la columna creada con ROW_NUMBER() nos dará el índice de las compras hechas, siendo que a cada cambio de cliente el conteo comenzará de nuevo y, con ello, seremos capaces de extraer la fila que queremos. La consulta para obtener el índice por cliente será la siguiente:

SELECT Cliente, Producto,
       Precio, ROW_NUMBER() OVER (PARTITION BY Cliente) 
       as row
FROM ventas

Resultando en:

Tabla de ventas con índice
Cliente Producto Precio row
Cliente_1 Pasta 25 1
Cliente_1 Agua 22 2
Cliente_3 Pasta 26 1
Cliente_3 Sopa 26 2
Cliente_3 Fruta 21 3
Cliente_5 Pasta 25 1

Este tipo de funciones son conocidas como funciones de ventana o window functions, y son especialmente útiles cuando queremos hacer un cálculo para ciertos grupos de filas, pero sin tener que hacer una agrupación directamente. Dicho de otra forma, nos permiten seccionar nuestros datos sin tener que perder el formato original por resumirlos al usar un GROUP BY.

Ahora, si queremos obtener la primera compra registrada por cliente, simplemente debemos tomar a nuestra consulta anterior como una subconsulta y filtrarla, tal que:

SELECT * FROM (
  SELECT Cliente, Producto,
       Precio, ROW_NUMBER() OVER (PARTITION BY Cliente) 
       as row
  FROM ventas
) a
WHERE row = 1

Lo que resultará en:

obtención de la primera observación
Cliente Producto Precio row
Cliente_1 Pasta 25 1
Cliente_3 Pasta 26 1
Cliente_5 Pasta 25 1

Este tipo de agrupaciones son extremadamente útiles cuando queremos buscar datos anómalos o específicos que se encuentran en una cierta posición (o posición estimada), respecto a otras columnas que podemos usar de referencia. Veamos un ejemplo más ilustrativo: Eliminar duplicados. Imagina que tenemos una tabla que guarda a todos los usuarios registrados en un sitio web, con datos clave como su correo electrónico, domicilio y número de teléfono. No existe ningún impedimento para que una misma persona se registre más de una vez, utilizando variaciones en su nombre o dirección, por lo que podemos hacer uso de su número de teléfono o correo electrónico para identificar usuarios únicos; En este caso nos decantaremos por el número de teléfono, ya que para un usuario es más complicado obtener un nuevo número antes que crearse un segundo correo electrónico. Supongamos que la tabla se llama usuarios_pagina y cuenta con las columnas ID, Nombre_usuario, telefono, correo_electronico y otras columnas con información adicional, por lo que nuestra consulta será la siguiente:

SELECT * FROM (
  SELECT ID, Nombre_usuario,
       telefono,correo_electronico,
       ROW_NUMBER() OVER (PARTITION BY telefono) row
  FROM usuarios_pagina
) a
WHERE a.row = 1

ROW_NUMBER() es solo una de varias funciones de ventana disponibles en SQL, junto con otras como RANK(), DENSE_RANK() o NTILE(). Cada una tiene usos específicos según el tipo de análisis que necesites hacer.

Aunque por ahora no profundizaremos en ellas, ya que este capítulo se enfoca en los fundamentos de SQL, es importante que sepas que existen y las tengas en mente para proyectos futuros donde necesites análisis más detallados sobre particiones de datos.

Supongamos ahora que tenemos una columna con fechas de registro denominada fecha_reg, con la cual podemos obtener al usuario único más reciente, ordenando los datos de la siguiente manera:

SELECT * FROM (
  SELECT ID, Nombre_usuario,
       telefono,correo_electronico, fecha_reg,
       ROW_NUMBER() OVER (PARTITION BY telefono ORDER BY fecha_reg DESC) row
  FROM usuarios_pagina
) a
WHERE a.row = 1

Esta consulta considera la observación más reciente al reordenar las fechas por número de teléfono en forma descendente. De manera inversa, si quisiéramos el registro más viejo, simplemente invertiríamos la orden DESC por ASC.

2.5.3.2 Filtros en una agrupación

Como es lógico, podemos filtrar datos que se encuentren agrupados a partir de columnas ya existentes en nuestros datos, por ejemplo: Utilizando la tabla ventas que analizamos hace un momento, podemos hacer un agrupamiento por clientes para obtener el total comprado por cada una, aplicando un filtro sencillo para ver el resultado del cliente 1, tal que:

SELECT Cliente, SUM(Precio) as Total
FROM ventas
WHERE Cliente = "Cliente_1"
GROUP BY Cliente

Resultando en:

Cliente Total
Cliente_1 47

Pero ¿y si quisiéramos filtrar la función de agregación en lugar del cliente? En ciertas ocasiones nos interesará saber qué usuarios (o valores de agrupación) cumplen con un cierto valor, por lo que filtraremos a partir de un cálculo realizado, sin embargo, el uso de la sentencia WHERE no puede ser aplicado en estos casos, ya que no estaríamos tomando directamente un valor ya existente de la tabla que estemos agrupando, por lo que se nos devolvería un error en caso de intentar usarlo. Para obtener los valores que cumplan esta condición, simplemente debemos usar la sentencia HAVING, por ejemplo, para filtrar sumas mayores o iguales 40:

SELECT Cliente, SUM(Precio) as Total
FROM ventas
GROUP BY Cliente
HAVING SUM(Precio) >= 40

Resultando en:

Cliente Total
Cliente_1 47
Cliente_3 73

Lógicamente, al filtrar filas que se crearon después de la agrupación, debemos establecer la sentencia HAVING después de GROUP BY.

Siguiendo la lógica anterior, podemos evitar el uso de HAVING y simplemente filtrar nuestra consulta convirtiéndola en una subconsulta, por ejemplo:

SELECT * FROM (
  SELECT Cliente, SUM(Precio) as Total
  FROM ventas
  GROUP BY Cliente
) a
WHERE a.Total >= 40

Lo que nos devolvería el mismo valor, pero un problema con esta forma de filtrado es el hecho de que se tiene que volver a leer la consulta creada originalmente para aplicarle un filtro, lo que en tablas muy grandes puede alentar el proceso de carga de la consulta, razón por la que el uso de HAVING es más recomendable en situaciones de este estilo.

2.6 Simplificando consultas con WITH: Expresiones comunes de tablas

A pesar de que el lenguaje SQL es bastante sencillo e intuitivo para extraer nuestros datos, no siempre vamos a querer reescribir consultas, sobre todo cuando estas implican una cantidad considerable de uniones de tablas que puedan volver el proceso algo engorroso y complicado de leer por otros usuarios. Una forma en la que podemos evitarnos mucha cantidad de querys complejos de leer es mediante la separación por pasos con WITH, convirtiendo cada sección individual en una CTE1, lo cual nos permitirá crear consultas más complejas, las que podrán ser reutilizadas para crear diferentes vistas únicamente llamándolas mediante un nombre identificador que nosotros asignemos. La idea es sencilla: Supongamos que tenemos una tabla de registros de usuarios en una plataforma llamada logs_users, en la cual cada fila equivale a un log de un usuario, por lo que cada usuario puede tener múltiples observaciones, siendo la primera observación cuando este último ingresó por primera vez a la plataforma, y la última correspondiente a su última visita (valga la redundancia), incluyendo además en cada fila el tiempo de conexión correspondiente a cada log. Supongamos ahora que tenemos otra tabla que únicamente contiene el id de los usuarios registrados y su nombre, llamada user_info, por lo que a partir de esas dos tablas queremos generar un reporte del último inicio de sesión de cada usuario, junto con el tiempo que estuvieron en línea, si aplica. Lógicamente, al querer saber la información de todos los usuarios registrados, tendremos que usar como tabla principal a user_info, ya que logs_users solo contiene información de aquellos usuarios que ya iniciaron sesión en alguna ocasión, y al usar la otra tabla entonces sabremos quienes de ellos aún no han usado la plataforma, lo que nos puede servir en un análisis a futuro. Ahora, regresando a la consulta, para calcular el tiempo que estuvieron los usuarios en su última sesión, podemos hacer uso de una función de ventana:

SELECT *
FROM (
  SELECT id, tiempo_sesion, fecha_sesion,
       ROW_NUMBER() OVER (PARTITION BY id 
                          ORDER BY fecha_sesion DESC) AS ult_fila
  FROM logs_users
) A
WHERE A.ult_fila = 1

Finalmente, para unir ambas tablas podemos hacer uso de un subquery, es decir:

SELECT uf.id, uf.user_name, ult_ses.tiempo_sesion, ult_ses.fecha_sesion
FROM user_info uf
LEFT JOIN (
  SELECT *
  FROM (
    SELECT user_id, tiempo_sesion, fecha_sesion,
         ROW_NUMBER() OVER (PARTITION BY user_id 
                            ORDER BY fecha_sesion DESC) AS ult_fila
    FROM logs_users
  ) A
  WHERE A.ult_fila = 1
) ult_ses ON ult_ses.user_id = uf.id

Hasta este punto el query puede parece lógico, sin embargo, este tipo de consultas pueden volverse complicadas y difíciles de manejar conforme vayamos aumentando la cantidad de uniones de tablas y columnas a utilizar. Para evitar que estas consultas se vuelvan complejas de leer, podemos hacer uso de las expresiones comunes de tablas para dividir los cálculos de cada vista en pasos separados. Comencemos por el query referente al cálculo del tiempo de sesión de los usuarios:

WITH logs_info_base AS (
  SELECT user_id, tiempo_sesion, fecha_sesion,
       ROW_NUMBER() OVER (PARTITION BY user_id 
                          ORDER BY fecha_sesion DESC) AS ult_fila
  FROM logs_users
)
SELECT *
FROM logs_info_base
WHERE ult_fila = 1

Donde simplemente englobamos el cálculo del último inicio de sesión en un paso anterior y le asignamos el nombre logs_info, para poder llamarlo después. Como siguiente paso, vamos a hacer el JOIN con la tabla principal:

WITH logs_info_base AS (
  SELECT user_id, tiempo_sesion, fecha_sesion,
         ROW_NUMBER() OVER (PARTITION BY user_id 
                            ORDER BY fecha_sesion DESC) AS ult_fila
  FROM logs_users
),
logs_info AS (
  SELECT * 
  FROM logs_info_base 
  WHERE ult_fila = 1
)
SELECT uf.id, uf.user_name,
       logs_info.tiempo_sesion,
       logs_info.fecha_sesion
FROM user_info uf
LEFT JOIN logs_info ON logs_info.user_id = uf.id;

Como puedes ver, en este query usamos dos CTE’s encadenadas, lo que implica que la segunda necesariamente depende de la creación de la primera para poder ejecutar la sentencia completa.

En este caso, usamos nombres similares para reflejar la secuencia lógica del proceso: primero generamos una tabla base con un número de fila por usuario (logs_info_base) y luego filtramos para obtener solo la fila más reciente de cada usuario (logs_info). Aunque aquí mantenemos nombres parecidos por claridad, en tus propios ejercicios puedes usar los nombres que te resulten más cómodos o descriptivos. Lo importante es que cada CTE tenga un nombre único, para evitar ambigüedades y errores en la consulta.

Este patrón de CTE’s encadenadas es muy útil cuando necesitas realizar transformaciones por pasos, permitiendo mantener el código más ordenado y legible. Más allá de su sintaxis, las CTE’s representan una forma de pensar en bloques lógicos, lo cual es esencialmente útil cuando analizamos bases de datos complejas o cuando necesitamos estructurar una transformación en varios pasos sin caer en subconsultas difíciles de leer.

¿Qué sigue ahora?

En la gran mayoría de los trabajos enfocados al análisis de datos, será normal que toda la información que utilicemos en nuestro día a día provenga de SQL, por lo que considero que esta sección es esencial para tu completo desarrollo en esta área, teniendo al menos una introducción de este lenguaje que te permita tener las herramientas suficientes para que puedas extraer todos los datos que vayas a utilizar, entendiendo de forma clara las consultas realizadas. Afortunadamente, el uso de SQL es bastante sencillo de entender, especialmente a un nivel básico o intermedio, y proporciona una base sólida para trabajar con datos de manera efectiva. Una vez que hayas dominado la realización de consultas en SQL, estás listo para sumergirte de lleno en el lenguaje R. Claro que este libro no tiene como objetivo el hacerte un experto de SQL, pero por el momento será más que suficiente para poder continuar con temas más complejos (en el área de datos). ¡Prepárate!, porque a partir de este punto nos adentraremos en la parte central de este libro, explorando las profundidades del análisis de datos con R.
