9  Presentando mis datos: Dashboards (Shiny) y apps web

Como no podía ser de otra manera, vamos a culminar este viaje en el entendimiento del análisis de datos con la creación de reportes presentables a otras personas. Cuando nosotros hacemos cálculos en los datos que hemos creado, creamos resúmenes de tablas, gráficos o incluso mapas, no podemos solo soltar esa información en crudo al púbico objetivo (coordinadores de otras áreas no relacionadas a datos, ejecutivos, o el público general). Siempre debemos buscar crear una visualización general de la información que estamos transmitiendo, procurando que esta sea amigable y entendible para las demás personas. Uno mismo como analista de datos entiende qué significan los cálculos realizados en un data frame, o los coeficientes devueltos por un modelo estadístico/modelo de ML, lo que es lógico al ser nosotros quienes creamos el proceso, pero ¿y las demás personas? ¿Cómo podemos validar, por ejemplo, ante un jefe de área, que un proyecto en la empresa no es viable?, ¿o que una cuenta tiene el potencial para duplicar los ingresos? Todas estas respuestas pueden ser respondidas con la creación de un dashboard (tablero) o un reporte.

¿Qué diferencia hay entre un dashboard (tablero) y un reporte? Básicamente, un dashboard tiene el propósito de mostrarnos múltiples métricas y datos fáciles de entender en una sola vista, lo que nos ayudará a tener un conocimiento más general de un negocio, área o tópico específico, mientras que un reporte será mucho más específico, procurando explicar ciertas métricas o resultados obtenidos de una forma mucho más meticulosa.

El capítulo final del presente libro tendrá como objetivo las siguientes dos tareas:

  1. Entender, crear, customizar y llevar al límite los dashboards creados mediante la librería shiny de R, así como explotar el funcionamiento de estos programas no solo con la creación de tableros de datos, si no para otras situaciones, como la automatización de tareas.
  2. Comprender cómo estructurar aplicaciones creadas con shiny, desde la organización de sus archivos hasta la construcción de interfaces, servidores, módulos, filtros, gráficos y tablas interactivas. Este libro no tiene como objetivo revisar únicamente ejemplos básicos, por lo que buscaremos entender cómo convertir nuestros códigos de R en aplicaciones funcionales, reutilizables y relativamente complejas.

Sin más que añadir a la presentación del capítulo final, ¡comencemos con ello!

9.1 Shiny

El método por excelencia para crear dashboards diréctamente en R es a través de la librería shiny, la cual nos brindará las herramientas no solo para crear el tablero que necesitemos, sino también para customizarlo, añadirle todas las funcionalidades que podamos imaginar, combinarlo con múltiples librerías e incluso con otros lenguajes de programación, como JavaScript.

Propiamente, la librería shiny no se enfoca únicamente en la creación de un dashboard, en realizad se utiliza para crear aplicaciones web. Una aplicación web no es más que un programa que, a partir de ciertos parámetros establecidos por el usuario, le devuelve un resultado en específico. Es importante que sepamos diferenciar a las aplicaciones web de dashboards como los creados en Power BI. Mientras que en este último solo podremos crear informes basados en datos y, como mucho, exportar los datos en crudo a un archivo excel, una aplicación web es más maleable y versatil. Con shiny no solo podemos crear dashbaords, también podremos crear procesos de automatización, vistas de modelos estadísticos, e inclusive creación en tiempo real de reportes en, por ejemplo, formatos PDF, excel, etc. El límite de uso para una aplicación web solo será el que te permita tu imaginación.

9.1.1 ¿Cómo funciona una Shiny app?

Una Shiny app se basa en inputs (información de entrada) y outputs (resultados), lo que dicho de otra manera, implica que a partir de los parámetros de entrada que el usuario introduzca (input), el programa va a generar los cálculos pertinentes en el backend1 para devolver un resultado (output), el cual puede ser de cualquier tipo, como una cadena de texto, un número, un gráfico, una tabla o una simple notificación (Shiny: Web Application Framework for r, n.d.). Gráficamente:

La app que construyamos siempre va a incluir estos tres elementos:

  1. Información global: Los datos que vamos a cargar, las librerías a utilizar y las opciones generales que vamos a establecer por default.
  2. UI: Conocido también como el User Interface o interfaz de usuario (conocida también como frontend), será el lienzo sobre el cual nosotros customizaremos el programa, le daremos estilo y mostraremos tanto los inputs como los outputs.
  3. Server: El servidor de nuestro programa o backend, donde se establecerá la lógica de los cálculos de nuestro programa, para poder mostrar los outputs en el UI.

A pesar de que estos elementos pueden ir en el mismo archivo o en archivos separados, siempre serán lo mínimo necesario para poder crear una aplicación web. Más adelante veremos elementos adicionales, cómo los módulos, llamado a códigos específicos, etc, pero por el momento nos centraremos en estos tres pilares fundamentales.

9.1.2 Comenzando a construir mi primer programa

Generalmente, el orden lógico para construir nuestro programa será:

\[ Global \rightarrow UI \rightarrow Server \]

En una sección global, típicamente comenzaremos cargando las librerías necesarias, por ejemplo:

library(shiny)
library(dplyr)

options(...)

datos_generales <- read.csv("misdatos.csv")

Para después pasar a diseñar la interfaz de usuario.

En ciertos casos tendremos que crear una carpeta adicional dentro del directorio del projecto, denominada www. Esta carpeta tendrá como objetivo el guardar elementos adicionales que después vamos a referenciar en la app, como pueden ser imágenes, logotipos, código HTML/CSS, etc. Más adelante veremos algunos ejemplos de uso de esta carpeta.

9.1.3 UI

Una interfaz de usuario típicamente estará contenida dentro de una función fluidPage(), la cual no es más que un contenedor basado en HTML. De hecho, si nosotros ejecutamos de forma separada la función, podremos notar que el resultado es código en HTML:

library(shiny)

as.character(fluidPage())
[1] "<div class=\"container-fluid\"></div>"

Usualmente las funciones que más vamos a utilizar para customizar nuestra interfaz son:

  • fluidPage(): Para crear páginas que se adapten al tamaño de la pantalla (ya sea de computadora o celular).

  • fixedPage(): Para crear páginas basadas en un tamaño fijo de pantalla.

  • navbarPage(): Para crear barras de navegación. Usualmente va dentro de funciones como fluidPage(), fixedPage() o bootstrapPage().

  • column(): Para crear contenedores div dentro de un contenedor (fluidPage(), por ejemplo) de tipo columna. Requiere establecer el tamaño de la columna mediante el argumento width, con valores que van de 1 a 12 (12 para abarcar toda la pantalla).

  • fluidRow(): Para crear filas en las que se pueden introducir de forma alineada columnas.

Existen otros tipos de contenedores más enfocados a dashbaords, los cuales estarán proporcionados por librerías como shinydashboard, shinydashboardPlus, bs4Dash, argonDash, entre otros, pero esas dependencias serán analizadas más adelante con detalle. Por ahora comenzaremos por dar un entendimiento a la forma en la que funciona la customización básica de una aplicación web.

Empecemos creando un contenedor simple con fluidPage(). Supongamos que en ese contenedor queremos introducir 4 columnas diferentes, por lo que, sabiendo que el ancho máximo de una columna es 12, entonces tendremos que dividir ese número entre 4 para poder generar esas columnas en la misma línea, es decir:

fluidPage(
  column(width = 3), column(width = 3), column(width = 3), column(width = 3)
)

Si en lugar de establecer una anchura igual a 3, esta fuera igual 4, entonces la última columna se enviaría debajo del resto de contenedores, esto ya que la suma de anchos de las primeras tres ya daría el valor máximo de 12.

Es posible anidar contenedores column() para crear subsecciones dentro de nuestro programa. En un caso en el tengamos una columna con un ancho de, por ejemplo, 6, si creamos otra columna dentro de la primera, esta última va a interpretar el espacio como su 100% de hoja, por lo que si nosotros establecemos un width = 12 en la columna secundaria, esta tendrá el tamaño total de la columna superior con width = 6 (manteniendo un margen de espaciado). Si estableciéramos el ancho de la columna secundaria en width = 6, esto sería el equivalente a un width = 3 para una columna exterior, debido al ancho del contenedor superior. Gráficamente:

El código en este caso es bastante sencillo, ya que solo escribiremos columnas dentro de otras:

fluidPage(
  column(
    width = 6,
    column(width = 6),
    column(width = 6)
  )
)

Además del espaciado y funciones básicas de HTML, nosotros también podemos aplicar estilo CSS a una aplicación web, por ejemplo: Por default, una columna siempre alineará los elementos contenidos dentro de ella, como textos o gráficos, a la izquierda, por lo que podemos aplicar una justificación con el argumento align = "center", es decir:

fluidPage(
  column(width = 6, align = "center")
)

Así como cambiar parámetros de estilo, como el color de fondo, por ejemplo:

fluidPage(
  column(width = 6, align = "center",
         style = "background-color: red")
)

Vayamos a un ejemplo más interesante: Supón que queremos crear una página donde el título se encuentre centrado hasta arriba, existan dos columnas y, dentro de la segunda columna, existan dos secciones, una debajo de la otra, es decir, lo visto en la figura 9.2. Para crear esta vista, simplemente tendremos que anidar los contenedores de tal forma que podamos establecer dicha vista. En este caso, las columnas diseñadas aplicadas de la siguiente forma:

fluidPage(
    column(width = 12, align = "center", h1("Título del programa")),
    column(width = 6, align = "center", p("Primer columna")),
    column(width = 6,
      column(
        width = 12, align = "center", p("Primer subcolumna"), 
        style = "background-color: #c96b73"
      ),
      column(
        width = 12, align = "center", p("Segunda subcolumna"),
        style = "background-color: #70c96b"
      )
    )
  )

Si nosotros cargamos esta interfaz en nuestro navegador (o el visualizador local de RStudio), el aspecto de esa vista será:

Nótese que, para introducir el título, usamos otro tipo de contenedor HTML h1(), mientras que para textos de cada columna, usamos un contenedor simple de texto p(). Además de lo anterior, también podemos notar que el texto de la segunda subcolumna se encuentra prácticamente pegado al de la primera, esto ocurre ya que, al no existir elementos dentro del espaciado entre ambas columnas, el contenedor de texto se introducirá en la parte superior. En cuanto añadamos algunos otros inputs y outputs, podremos comenzar a ver la diferencia entre espacios de cada contenedor.

Como ya te habrás dado cuenta, una ventaja notable del uso de shiny para crear apps web es la facilidad para crear una interfaz muy rápidamente. Si nosotros tradujéramos este código a HTML, tendríamos:

9.1.3.1 Inputs

Una vez establecido el diseño de nuestra app, podemos comenzar a introducir los elementos con los que el usuario va a interactuar, como lo son selectores y botones. Estos inputs serán una forma de interactuar dinámicamente con las funciones del servidor. Los principales inputs que vamos a utilizar serán:

  • selectInput(): Para seleccionar un elemento de una lista desplegable.
  • numericInput(): Para introducir valores numéricos como inputs.
  • textInput(): Para introducir cadenas de texto como inputs.
  • sliderInput(): Para introducir valores numéricos dentro de un rango establecido.
  • dateInput(): Para introducir fechas individuales.
  • dateRangeInput(): Para introducir rangos de fechas (fecha inicial y fecha final).
  • fileInput(): Para subir archivos como inputs (Ej. Documentos excel o csv).
  • checkboxInput(): Para retornar valores booleanos como inputs.
  • radioButtons(): Para seleccionar un elemento concreto. Recomendable en lugar de un selectInput() cuando tenemos pocos elementos a escoger.
  • actionButton(): Para retrasar la renderización de un output hasta dar click a un evento (botón).

Estos inputs son las versiones nativas de la librería shiny, sin embargo, existen versiones mucho más customizables y amigables a la vista, provenientes de la librería shinyWidgets, por ejemplo: pickerInput() como equivalente de selectInput(), o airDatepickerInput() como equivalente de dateInput() y dateRangeInput() (si tenemos el parámetro range = TRUE), entre otros elementos. Poco a poco iremos viendo cada uno de ellos.

Ahora, ¿cómo usamos estos inputs para interactuar con el servidor? Dependiendo de cada input, se requerirán ciertos parámetros específicos, por ejemplo, en el caso de un selectInput(), tendremos que añadir las opciones a escoger dentro de un vector, mientras que un textInput() no requerirá que añadamos necesariamente ninguna opción (aunque si podemos establecer una sugerencia predefinida), ya que este último tiene la intención de aceptar un texto libre como input. A pesar de esas diferencias, todos los inputs necesariamente requerirán un elemento: El InputID. Un InputID no es más que un nombre único que utilizaremos como identificador para llamar del lado del servidor y que, en consecuencia, permita obtener correctamente las selecciones o parámetros que el usuario haga con ese input específico, por ejemplo:

selectInput(
  inputId = "mis_opciones",
  label = "Opciones",
  choices = c("Opcion 1", "Opcion 2", "Opcion 3")
)

En este caso, para nosotros poder llamar a este input en el lado del servidor, lo haremos mediante el código input$mis_opciones. Necesariamente debemos hacer uso de input$ para seleccionar el objeto en cuestión, ya que de esta forma el programa identifica que se trata de un input dinámico y no de un objeto preestablecido en la sesión.

Los inputs (dependiendo del caso) devuelven las respuestas seleccionadas como un vector, por lo que si hay una selección múltiple en alguno de ellos, lógicamente podremos acceder a cada uno de sus elementos mediante indexación, por ejemplo, input$mis_opciones[1]. Más adelante veremos las implicaciones de vectores múltiples del lado del servidor y cómo trabajarlos correctamente.

9.1.3.2 Outputs

Así como los inputs son solo una cara de la moneda, siendo aquello que el usuario introduce como parámetro(s), podemos definir a los outputs como aquellos elementos que serán retornados a partir de la información establecida, siendo resultados dinámicos con los que podremos interactuar en mayor o menor medida, dependiendo del tipo de elemento devuelto. Dicho en otras palabras, un output no es mas que el objeto resultante de todo el procesamiento de información realizada en el backend, a partir de los parámetros que se hayan introducido. Típicamente, los outputs en aplicaciones web suelen ser: cadenas de texto, gráficos, mapas y elementos interactivos (outputs de los que podemos obtener más información), y estarán en función de lo que queramos mostrar en nuestro programa.

Estos objetos resultantes son mucho más sencillos de entender dentro del UI que sus contrapartes (inputs), ya que suelen estar basados únicamente en tres parámetros: OutputID (identificador único del elemento resultante que queremos mostrar), width (anchura del elemento) y height (altura del elemento). Veamos un ejemplo:

fluidPage(
  column(
    width = 3,
    selectInput(
      inputId = "mis_opciones",
      label = "Opciones",
      choices = c("Opcion 1", "Opcion 2", "Opcion 3")
      )
  ),
  column(
    width = 9,
    plotOutput("mi_grafico", width = "400px", height = "600px")
  )
)

Del código mostrado anteriormente, podemos darnos cuenta de que cada elemento, ya sean inputs o outputs, tiene un identificador único, con el que podremos interactuar con esos elementos en particular, pero ¿cómo exactamente logramos que las entradas y salidas de nuestro programa se conecten entre si? Aquí es donde entra en juego el corazón de nuestro programa: El servidor o server.

9.1.4 Server

Hasta ahora, hemos visto rápidamente cómo funciona una interfaz en shiny, así como el manejo que podemos hacer para customizar la aplicación que el usuario final va a ver en su computadora, pero esta sección nos ayudará a entender cómo funciona la magía en nuestra programa, es decir, la forma en la que haremos que los parámetros establecidos nos den un resultado específico. Al principio será un tema bastante sencillo, pero poco a poco se irá complicando, por lo que procuraré ser lo más simple posible en las explicaciones de la lógica utilizada en esta parte.

Primero, comencemos por entender el servidor en si. Un código o script en R no es interactivo en si, ya que necesitamos establecer objetos y valores para obtener resultados, ¿cómo podemos hacerlo dinámico? Simplemente convirtiendo ese script en una función, donde los objetos y valores sean parámetros de ella. El servidor de una aplicación web no es más que una función con dos parámetros: input y output, principalmente. Habrá aplicaciones en las que establezcamos un tercer parámetro: session, el cual veremos con detalle más adelante, cuando profundicemos en la reactividad del servidor.

Típicamente, el servidor se verá de la siguiente manera:

server <- function(input, output){
  
}

9.1.4.1 Conectando la interfaz: Reactividad

Como recordarás, anteriormente te mencioné que la forma en la que nosotros podremos acceder a un elemento de un input o un output será mediante su ID, al cual llamaremos con un signo $. Regresemos al ejemplo anterior, es decir:

fluidPage(
  column(
    width = 3,
    selectInput(
      inputId = "mis_opciones",
      label = "Opciones",
      choices = c("Opcion 1", "Opcion 2", "Opcion 3")
      )
  ),
  column(
    width = 9,
    plotOutput("mi_grafico", width = "400px", height = "600px")
  )
)

Obviamente, la idea de esa interfaz es mostrar un gráfico (identificado por el ID mi_grafico) que estará en función de una de las opciones del input mis_opciones. Para nosotros poder referirnos a ese gráfico en el lado del servidor, vamos a crear un objeto que lo renderice con la función renderPlot(), tal que:

server <- function(input, output){
  
  output$mi_grafico <- renderPlot({
    plot(x, y)
  })
  
}

Nótese que, además del uso del paréntesis, también utilicé corchetes dentro de la función renderPlot(), ya que eso me permitirá una mayor flexibilidad al momento de escribir códigos más complejos cuando quiera renderizar otro tipo objeto.

En de la función renderPlot() es donde vamos a hacer uso de funciones menos dinámicas para generar el gráfico, siendo en este caso plot(). Todas las librerías de gráficos, mapas y objetos interactivos que esten habilitadas para trabajar con aplicaciones web siempre tendrán una función con el prefijo render, la cual será aquella que estableceremos en el servidor para generar el objeto que necesitemos. Ahora, las funciones de estilo render son elementos reactivos, es decir, aquellos que cambian su valor dependiendo de los elementos de entrada (inputs), como lo mencioné anteriormente.

Supongamos que de ese conjunto de datos queremos visualizar el promedio de población registrada en 2011 (population_2011) por región de la localidad rgn11nm. Esa tarea es en realidad muy sencilla, ya que simplemente podemos crear un gráfico con ggplot2 que nos muestre la información que queremos, tal que:

uk_data |> 
  group_by(rgn11nm) |> 
  summarise(promedio = mean(population_2011, na.rm = T)) |> 
  na.omit() |> 
  arrange(desc(promedio)) |> 
  ggplot(aes(x = promedio, y = forcats::fct_reorder(rgn11nm, promedio) ))+
  geom_col(fill = "darkred") + 
  theme_minimal() +

La parte interesante comienza cuando queremos realizar múltiples vistas accionables por un usuario ajeno a nuestro código. Supongamos que queremos filtrar nuestros datos dinámicamente por tipo de ciudad (size_flag), aquí tenemos dos opciones: Generar un filtro que acepte solo un argumento (==) o un filtro múltiple (%in%). Comencemos con un filtro de un solo argumento, el cual vamos a implementar en una app que nos permita realizar esta operación a partir de un selectInput(). Primero, nuestro selector necesitará obtener las opciones de tamaños de ciudad disponibles en nuestros datos, por lo que podemos hacer uso de una función unique(), tal que:

selectInput(inputId = "ciudad", label = "Tipo de ciudad", 
            choices = unique(uk_data$size_flag))

Además, podemos reordenar los valores por orden alfabético:

selectInput(inputId = "ciudad", label = "Tipo de ciudad", 
            choices = sort(unique(uk_data$size_flag)))

El inputId del selector es ciudad, por lo que para aplicar el filtro dentro del código del gráfico, tendremos que llamarlo en una función filter() de la siguiente manera:

uk_data |> 
  filter(size_flag == input$ciudad) |>
  group_by(rgn11nm) |> 
  summarise(promedio = mean(population_2011, na.rm = T)) |> 
  na.omit() |> 
  arrange(desc(promedio)) |> 
  ggplot(aes(x = promedio, 
             y = forcats::fct_reorder(rgn11nm, promedio) ))+
  geom_col(fill = "darkred") + 
  theme_minimal() +
  ylab(label = "Region")

9.1.4.3 Objetos reactive()

Los objetos reactive() son una forma en la que nosotros convertiremos nuestros datos a una función dinámica, la cual usaremos después para generar nuestros outputs. Este tipo de elementos son convenientes cuando queremos ramificar una cierta operación en múltiples elementos de salida, por ejemplo: Supongamos que queremos generar dos gráficos diferentes que estarán ligados al mismo filtro. Podríamos directamente generar el código en cada uno de los objetos render, escribiendo en cada caso el filtro que necesitamos, sin embargo, si queremos escalar este programa, añadiendo cada vez más y más elementos de salida, individualizar esta operación a cada caso hará que nuestro programa se vuelva muy lento. Una forma más eficiente de ejecutar estas operaciones será partiendo de un objeto reactive() que ya filtre nuestros datos, para después únicamente llamarlo en cada render. Gráficamente:

Regresemos a nuestro ejemplo anterior. Vamos a añadir un gráfico adicional que ahora mida el porcentaje de residentes que tienen ingresos altos (arriba de 10,000), divido por zonas con/sin universidades, es decir:

9.1.4.4 Eventos reactivos

Cuando seleccionamos ciertas opciones, no siempre queremos que estas se ejecuten de inmediato. Piensa en una aplicación que tiene tres, cinco o hasta más inputs con los que generar un gráfico o elemento cualquiera. Seleccionar todos los parámetros nos va a llevar un tiempo, por lo que para un usuario que esté usando el programa, será bastante incómodo tener que ver múltiples renderizaciones del objeto mientras termina de seleccionar todos los parámetros, por lo que en esa situación lo más óptimo sería utilizar un boton de acción que utilizaremos como evento, para que el objeto de salida sea ejecutado únicamente cuando ocurra dicho evento (el botón sea precionado). Una vez más, regresemos a la aplicación que hemos creado hasta ahora, supongamos que queremos renderizar ambos gráficos hasta que presionemos un botón, tal que:

actionButton(inputId = "graficar", label = "Generar gráficos")

Podemos añadir dicho boton en la interfaz de nuestra app, debajo del selector:

fluidPage(
    column(
      width = 12,
      align = "center",
      h1("Mi primer shiny app"),
      br(),
      selectInput("ciudad", "Tipo de ciudad", 
                  choices = unique(uk_data$size_flag)),
      actionButton(inputId = "graficar", label = "Generar gráficos")
    ),
    column(
      width = 6, align = "center",
      plotOutput("grafico", width = "600px")
    ),
    column(
      width = 6, align = "center",
      plotOutput("grafico_2", width = "600px")
    )
  )

Para ligar los gráficos a la acción de presionar el botón, solo debemos englobarlos en una función observeEvent():

observeEvent(input$graficar,{
  
  output$grafico <- renderPlot({
      
      datos_dinamicos() |> 
        group_by(rgn11nm) |> 
        summarise(promedio = mean(population_2011, na.rm = T)) |> 
        na.omit() |> 
        arrange(desc(promedio)) |> 
        ggplot(aes(x = promedio, 
                   y = forcats::fct_reorder(rgn11nm, promedio) ))+
        geom_col(fill = "#ad3d42") + 
        theme_minimal() +
        ylab(label = "Region")
      
    })
    
  output$grafico_2 <- renderPlot({
      
      datos_dinamicos() |> 
        group_by(rgn11nm, university_flag) |> 
        summarise(
          promedio_ing = 
            mean(activity_at_age_19_employment_with_earnings_above_10_000, 
                 na.rm = T)) |> 
        na.omit() |> 
        ggplot(aes(y = promedio_ing, x = rgn11nm, fill = university_flag)) +
        geom_col(position = position_dodge(width = 0.9), width = 0.7) + 
        theme_minimal()+
        theme(legend.position = "bottom",
              axis.text=element_text(size=7))+
        scale_fill_manual(values = c("#ad3d42", "#3d4ead")) +
        labs(
          x = "Zona",
          y = "Promedio")
      
    })
})

El uso de observeEvent() no está limitado a un solo elemento. Sencillamente podemos ligar la renderización de un output a múltiples accionables, añadiéndolos a una lista, por ejemplo:

observeEvent(list(input$graficar, input$otro_evento),{
  
  ...
  # Procesos ligados a múltiples eventos
  ...
  
})

El hecho de que un proceso se encuentre relacionado a múltipes eventos no significa que todos tengan que ocurrir, si no que al menos uno de ellos se ejecute, sea cual sea este.

Los eventos comunmente están asociados a botones, sin embargo, no es necesario que este sea el caso. De igual forma pueden estar ligados a ciertos inputs de otras clases, como selectores o booleanos. Todo dependerá del contexto de la aplicación y lo que necesitemos para mejorar la experiencia del usuario final.

Ahora, debemos considerar que estos eventos suponen que, el accionar el botón o input en nuestro programa, mantendrá la renderización activa, es decir, que al nosotros utilizar el botón, cuando volvamos a seleccionar parámetros en el programa, ya no será necesario volver a presionarlo, ya que el evento esperado ya ocurrió, por lo que ahora los gráficos se van a ejecutar con normalidad, sin estar ligados a nada. Cuando nosotros queremos asegurarnos de que un objeto únicamente sea renderizado cada que el elemento de acción es ejecutado, independientemente de las veces que hagamos esto, tendremos que utilizar bindEvent() en lugar de observeEvent(). Para el uso de bindEvent(), en lugar de englobar las funciones de renderización, las vamos a concatenar:

    output$grafico <- renderPlot({
        
        datos_dinamicos() |> 
          group_by(rgn11nm) |> 
          summarise(promedio = mean(population_2011, na.rm = T)) |> 
          na.omit() |> 
          arrange(desc(promedio)) |> 
          ggplot(aes(x = promedio, 
                     y = forcats::fct_reorder(rgn11nm, promedio) ))+
          geom_col(fill = "#ad3d42") + 
          theme_minimal() +
          ylab(label = "Region")
        
      }) |> 
      bindEvent(input$graficar)

De esta forma, no importa que ya hayamos ejecutado el evento, para cada nueva renderización del gráfico tendremos que volver a accionarlo.

9.1.4.5 La sesion

A la función que creamos para el servidor de nuestra aplicación también podemos añadirle un tercer argumento: session, siendo el código:

server = function(input, output, session){
  ...
}

El elemento session nos permitirá acceder al estado individual de cada sesión (valga la redundancia) creada por la aplicación, es decir, cuando nosotros ejecutamos una shiny app, cada vez que abrimos el programa en una pestaña diferente del navegador, esta pestaña representará una sesión individual que responderá únicamente a esa vista, siendo el argumento session una forma de acceder a esa información individual e interacturar con ella.

No siempre vamos a querer partir del estado original de una aplicación cuando la ejecutemos de nuevo, sobre todo cuando hay una cantidad considerable de parámetros que ya hemos establecido en el análisis que estemos haciendo. Podemos hacer uso de session para, por ejemplo, guardar ciertos parámetros dentro del link de nuestra app y, de manera automática, inicializarla con las opciones que hayamos dejado al finalizar la sesión anterior. Retomemos la aplicación que hemos creado hasta ahora, con el argumento session añadido en el servidor; Para poder inicializarla desde un punto específico, simplemente debemos utilizar el argumento enableBookmarking, es decir:

shinyApp(ui = frontend, server = backend, enableBookmarking = "url")

Para acceder al link exacto del estado actual de la aplicación, podemos hacer uso del input bookmarkButton() (añadiéndolo en el UI), que nos permitirá copiar ese link al portapapeles.

El uso de session no solo se limita a la información de la sesión actual, si no también a la manipulación dinámica de ella, por ejemplo, con los inputs. Pensemos en el input que utilizamos en la app de muestra para filtrar datos, este contiene las siguientes opciones:

9.1.4.6 Invalidando la reactividad

No en todas las situaciones en las que queramos ejecutar un evento vamos a necesitar que esta acción ocurra invariablemente (o que ocurra de la misma forma), ya que podemos tener ciertos inputs con valores incompatibles, o sin los valores necesarios para ejecutar de la forma correcta el output requerido. Regresemos al ejemplo que hemos venido trabajando hasta ahora, donde tenemos el input selectInput():

selectInput("ciudad", "Tipo de ciudad", choices = unique(uk_data$size_flag))

Al seleccionar datos individuales, este input por default siempre tendrá un valor asignado, sin embargo, si modificáramos el párametro multiple = T para aceptar más de un valor (manteniendo vacío el parámetro selected):

selectInput("ciudad", "Tipo de ciudad", choices = unique(uk_data$size_flag),
            multiple = T)

Esto permitirá que comencemos nuestra app sin un valor seleccionado por default. Obviamente, al no tener un valor establecido, el gráfico conectado a este input no se renderizará, ya que la amplitud del vector a filtrar será igual a cero, por lo que en esta situación podríamos invalidar la renderización de los gráficos en cuestión, esto haciendo uso de las funciones validate() (para generar un validador previo que evitará la renderización del objeto) y need() (para especificar el input a evaluar). El uso de estas funciones es bastante sencillo, siendo que simplemente debemos aplicarlas como un prefijo dentro de la función de renderización del output, por ejemplo, con output$grafico_2:

output$grafico_2 <- renderPlot({
  
      validate(
        need(input$ciudad, message = "Escoge un valor!")
      )
      
      datos_dinamicos() |> 
        group_by(rgn11nm, university_flag) |> 
        summarise(
          promedio_ing = 
            mean(activity_at_age_19_employment_with_earnings_above_10_000, 
                 na.rm = T)) |> 
        na.omit() |> 
        ggplot(aes(y = promedio_ing, x = rgn11nm, fill = university_flag)) +
        geom_col(position = position_dodge(width = 0.9), width = 0.7) + 
        theme_minimal()+
        theme(legend.position = "bottom",
              axis.text=element_text(size=7))+
        scale_fill_manual(values = c("#ad3d42", "#3d4ead")) +
        labs(
          x = "Zona",
          y = "Promedio")
      
    })

Por lo que, si no tenemos un valor establecido en input$ciudad, en lugar de devolver un error, se nos devolverá un mensaje de alerta preestablecido por nosotros:

Sencillamente, también podemos aplicar operaciones en una función need(), cuando queremos ser más específicos en las situaciones en las que queremos que no se renderice un objeto, por ejemplo:

validate(
  need(input$ciudad == "Small Towns", 
       message = "Esta selección es inválida. Escoge otra")
)

De esta forma, cuando se seleccione un valor que cumpla con esa condición, a pesar de tener un elemento en el vector del input, igualmente nos devolverá un mensaje de alerta en lugar del gráfico a renderizar.

Cuidado

No olvides que un selector múltiple generará un vector de longitud >=1, por lo que necesitarás cambiar el operador de filtro en el objeto reactive() de == a %in%, en este caso:

datos_dinamicos <- reactive({
    uk_data |> 
      filter(size_flag %in% input$ciudad)
  })

Usar validate() no es la única manera en la que podemos cancelar la renderización de un elemento, ni la más agradable visualmente, pero si la más sencilla. Veamos algunas formas más interesantes de abordar este problema, utilizando la librería shinypop (del siguiente reporsitorio de Github). Este paquete es una recopilación de funciones escritas en javaScript para mostrar notificaciones y mensajes de alerta, por lo que podemos utilizar estos elementos a nuestro favor para manejar la reactividad.

Comencemos con un ejemplo sencillo: Vamos a evaluar la renderización de los gráficos en la app que estamos construyendo, solo que en esta ocasión lo haremos mediante una estructura if, tal que:

observeEvent(input$graficar,{
    
    if (is.null(input$ciudad)==F){
      
      output$grafico <- renderPlot({
      ...
    })
    
    output$grafico_2 <- renderPlot({
      ...
    })
    
    }
  })

En este caso, si no seleccionamos un valor para input$ciudad lógicamente el vector resultante será de longitud cero, equivalente a un objeto NULL, por lo que al no cumplir esa condición, los gráficos no se crearán y no se devolverá un mensaje de error. Ahora, lo que si queremos que se devuelva es un mensaje de alerta emergente, por lo que podemos añadir a la evaluación una función de la librería shinypop, como puede ser noty(text = "El input 'ciudad' está vacío. Añade un valor", type = "warning"), resultando en:

observeEvent(input$graficar,{
    
    if (is.null(input$ciudad)==F){
      
      output$grafico <- renderPlot({
      ...
    })
    
    output$grafico_2 <- renderPlot({
      ...
    })
    
    }else{
      noty(text = "El input 'ciudad' está vacío. Añade un valor", 
           type = "warning")
    }
  })

En este caso, para poder visualizar la notificación emergente, tendremos que añadir el código use_noty() en la parte inicial de la interfaz, tal que:

fluidPage(
  use_noty(),
  ...

9.1.6 Subidas y bajadas de archivos

Los inputs y outputs no necesariamente deben limitarse a cadenas de texto, números o fechas simples, estos también pueden ser un archivo completo, desde archivos .xlsx, .csv, hasta documentos .pdf, entre muchos otros formatos. Del lado del input, el poder cargar un archivo en una shiny app nos permite una mucho mayor flexibilidad para crear ciertos análisis, aunque eso también implica que debemos ser cuidadosos respecto al tipo de archivos que permitiremos que el usuario suba, ya que no en todos los casos van a funcionar de la manera más óptima. Del lado del output la cosa se vuelve más sencilla, ya que la descarga de documentos estará más controlada, con el código que establezcamos para generar el documento, lo que nos permitirá minimizar errores en este lado del proceso.

9.1.6.1 File inputs

Subir un documento a una shiny app es bastante sencillo, ya que solo necesitaremos aplicar una función fileInput(), por ejemplo:

fileInput(inputId = "archivo_a_subir", 
          label = "Sube tu archivo", 
          placeholder = "doc.xlsx")

Al momento de nosotros subir un documento, este se va a interpretar como un input que se va a ramificar en varios elementos:

  • datapath: Ruta temporal del archivo.
  • name: Nombre original del archivo.
  • size: Peso total del archivo en bytes.
  • type: Tipo de archivo.

Cada uno de elementos nos serán útiles en diferentes situaciones que veremos a continuación, pero principalmente utilizaremos el elemento datapath, ya que con este vamos a leer el documento subido mediante otras funciones, como pueden ser read_excel(), read.csv(), etc.

Vamos a crear un ejemplo sencillo de aplicación para subir documentos. En este ejemplo vamos a utilizar la librería reactable (Reactable: Interactive Data Tables for r, n.d.) para crear una tabla condicional basada en el archivo que utilicemos como input. Comencemos por la interfaz:

library(reactable)

frontend <- fluidPage(
    column(
      width = 12, align = "center",
      h1("App para subir archivos")),
    sidebarPanel(
      width = 3,
      align = "center",
      fileInput("archivo", "Archivo a subir"),
      numericInput("filas", "Número de filas a leer", value = 999),
      selectInput("columnas", "Columnas a seleccionar", multiple = T,
                  choices = NULL)),
    column(
      width = 9,
      reactableOutput("tabla_dinamica")
    )
  )

Ahora, ¿qué queremos que haga nuestra aplicación? Primero, el input filas va a determinar la cantidad de filas que vamos a leer de nuestro archivo, siendo que con dicho archivo vamos a actualizar las selecciones del input columnas, para que ahora nos aparezcan todas las columnas existentes del documento de forma condicional, lo que nos permitirá seleccionarlas para visualizar el output tabla_dinamica. Vayamos por partes: El documento que vayamos a subir, sea cual sea, será el evento detonante para que, primero, el selector de columnas obtenga valores y, posterior a esto, podamos visualizar la tabla renderizada con las columnas seleccionadas. El primer paso lógico será convertir el documento a subir en un elemento reactivo, es decir:

data <- reactive({
      data <- readxl::read_excel(input$archivo$datapath, 
                                 n_max = input$filas)
      data
    })

Donde utilizamos input$filas para limitar la cantidad de filas retornadas.

Lógicamente, input$archivo es el evento que nos permitirá actualizar las opciones y la tabla, por lo que vamos a englobar dichos elementos en un observeEvent(), tal que:

observeEvent(input$archivo,{
      
      updateSelectInput(session = session,
                        "columnas", 
                        choices = colnames(data()),
                        selected = colnames(data()))
      
      output$tabla_dinamica <- renderReactable({
        data() |> 
          select(one_of(input$columnas)) |> 
          reactable()
      })
      
    })

Donde simplemente utilizamos el dato reactivo que cargamos en nuestra app para obtener y seleccionar los nombres de las columnas, mientras que la renderización de la tabla únicamente se va a basar en estas selecciones, por lo que necesariamente deberá estar englobada dentro del mismo evento.

Dados los pasos anteriores, el servidor de nuestra se conformará simplemente de ambos elementos, de la siguiente manera:

backend <- function(input, output, session){
    
    data <- reactive({
      data <- readxl::read_excel(input$archivo$datapath, 
                                 n_max = input$filas)
      data
    })
    
    observeEvent(input$archivo,{
      
      updateSelectInput(session,"columnas", 
                        choices = colnames(data()),
                        selected = colnames(data()))
      
      output$tabla_dinamica <- renderReactable({
        validate(
          need(input$columnas, 
               message = "Selecciona al menos una columa"))
        data() |> 
          select(one_of(input$columnas)) |> 
          reactable()
      })
    })
    
  }

Esta app nos permite leer cualquier archivo de tipo excel, pero esto invariablemente nos conducirá a un pequeño problema: ¿Qué pasaría si el usuario intentara subir otro tipo de archivo? Por ejemplo, un documento .csv. La función read_excel() es imcompatible con este tipo de archivos, por lo que podemos plantear dos soluciones a esta problemática:

  1. Aplicar una función condicional tryCatch(), para que, en caso de fallar con la lectura de un archivo .csv mediante read_excel(), utilice read.csv() en consecuencia. Para ello, simplemente debemos establecer esta última como función a ejecutar en caso de un error en la lectura del documento, siendo:

    data <- reactive({
    data <- 
    tryCatch({
        readxl::read_excel(input$archivo$datapath, 
                           n_max = input$filas)
    }, error = function(e){
       read.csv(input$archivo$datapath, 
                nrows = input$filas)
    })
  data
})

    Sin embargo, esta opción aún da pie a que el usuario introduzca un archivo ajeno a documentos de tipo .csv o .xlsx, generando otro error en el programa, y obviamente lo último que queremos hacer es crear un tryCatch() anidado, por lo que esta no es la solución más óptima.

  2. Limitar las opciones de subida a solamente aquellos documentos con terminación .csv o .xlsx, es decir:

    fileInput("archivo", "Archivo a subir", accept = ".xlsx")

    O bien, añadir más de un tipo de archivo disponible a subir, usando un vector como argumento c(".xlsx", ".csv").

Tip

Ambas soluciones son igual de validas, por lo que para escoger la más óptima, es recomendable pensar a partir de los posibles errores que pueda cometer el usuario final en cada caso, siendo la mejor opción la que de pie a menos problemáticas, sobre todo al escalar nuestra app.

Ahora, ¿hay un tamaño máximo permitido para los archivos cargados? Si. Por default, las apps en shiny solo permiten subir archivos no mayores a cinco megabytes (expresado en su equivalente en bytes: cinco millones), número que podemos configurar en el servidor de nuestra app o en la sección global, mediante la opción shiny.maxRequestSize, siendo en este caso:

options(shiny.maxRequestSize = 30000000)

El número máximo estará expresado en bytes, por lo que para subir un documento de, por ejemplo, treinta megabytes, será necesario establecer treinta millones de bytes, ya sea que lo hagamos con el número directo o simplificándolo con una multiplicación 30*1000^2.

9.1.6.2 Descargas

La descarga de archivos en apps shiny no es más que una forma dinámica de obtenerlos, mediante una función downloadHandler() llamada por un input downloadButton() o downloadLink(), con la que ejecutaremos funciones más sencillas de exportación, como pueden ser write.csv(), writexl::write_xlsx(), e incluso algunas basadas en formatos mucho más especializados, como lo son ggplot2::ggsave() o openxlsx::saveWorkbook(). Esta sección es especialmente interesante, ya que la descarga de archivos es considerablemente más sencilla que la subida de estos, además de que tenemos un control mucho mayor del resultado, por lo que podremos customizarlo de tal forma que sea muy agradable para el usuario final.

Comencemos con un ejemplo básico: Exportar los datos visualizados de nuestra app en una tabla resumida. Para este ejemplo utilizaremos el set de datos cheeses.csv, disponible en el repositorio oficial del libro, siendo:

Además de estos selectores, vamos a añadir un botón de descarga con un ID igual a descarga, objeto que representará el evento con el que importaremos los datos en un formato .xlsx. La interfaz en cuestión será:

resumen <- reactive({
      data |> 
        filter(color %in% input$color) |> 
        group_by_at(input$agrupacion) |>
        summarise(
          Conteo = n()
        ) |> 
        arrange(desc(Conteo))
    })

Para obtener estos datos en un documento .xlsx, utilizaremos la función downloadHandler(), la cual tendrá como argumento dos funciones anónimas: La primer función se encargará asignar un nombre al documento a partir de una cadena de caracteres, mientras que la segunda se encargará de llamar a la función generadora del documento (writexl::write_xlsx() en este caso), utilizando como argumento el nombre generado en la función anterior. El código en cuestión será:

output$descarga <- downloadHandler(
      filename = function(){
        paste0("resumen por ",input$agrupacion,".xlsx")
      },
      content = function(file){
        writexl::write_xlsx(
          x = list(
            Datos = resumen()
          ),
          path = file
        )
      }
    )

Donde los inputs también pueden ser usados como argumentos para generar el nombre del archivo de una forma muy sencilla.

backend <- function(input, output){

9.1.7 Customización de interfaces con HTML y CSS

En este punto ya tenemos bastante claro que una shiny app no es más que una forma de expresar código HTML, CSS y JavaScript mediante funciones de R, lo que hace lógico que podamos implementar elementos diréctamente escritos en estos lenguajes para poder customizar de manera más profunda nuestras apps. En esta sección nos enfocaremos meramente en HTML y CSS, ya que el uso de JavaScript en los programas que realicemos estará más relacionado a aspectos de funcionamiento de los widgets (sesión, interactividad y eventos).

9.1.7.1 HTML

Como ya vimos con anterioridad, las funciones dedicadas a la interfaz, como fluidPage(), son una forma de llamar contenedores de elementos, por ejemplo:

fluidPage(
  column(
    width = 10
  )
)

Equivalente en lenguaje HTML a:

<div class="container-fluid">
  <div class="col-sm-10"></div>
</div>

Donde class hace referencia a ciertos parámetros asignados a ese contenedor, que le darán una apariencia por default (la cual estará establecida por un estilo escrito en lenguaje CSS).

Todos estos elementos, sin importar la clase, son conocidos como tags. Dada la denominación anterior, una forma sencilla en la que podemos acceder a dichos elementos dentro de nuestra shiny app es llamándolos a partir del objeto del mismo nombre, por ejemplo:

tags$a()

Usualmente, cuando queremos establecer algunos parámetros de customización, establecer la codificación de la página (como UTF-8) o referenciar código adicional ubicado en otro archivo, haremos uso del tag head(), con el que asignaremos información que se va a aplicar a todo nuestro programa (de manera similar a la función options(), con la cual establecemos opciones en una sesión de R). Este tag debe ser escrito al principio de nuestro contenedor principal (fluidPage() en la mayoría de los casos), en la interfaz de la shiny app, por ejemplo:

fluidPage(
  tags$head(
    tags$meta(charset = "UTF-8"),
    tags$link(rel = "stylesheet", type = "text/css", href = "estilo.css")
  ),
  ...
)

Siendo que en este caso, utilizando el tag meta, establecimos la codificación que seguirán las cadenas de texto y demás elementos en nuestra interfaz, y mediante el tag link hacemos referencia a un archivo adicional que contiene parámetros CSS que nos ayudarán a customizar nuestra app. En casos en los que tenemos programas demasiado grandes, los cuales cuentan con una cantidad considerable de páneles y/o elementos específicos, es preferible separar el parámetro style en un archivo separado, lo que nos permitirá tener un código mucho más limpio. Si bien es posible establecer el código CSS en la sección head() mediante el tag style(), por ejemplo:

fluidPage(
  tags$head(
    tags$style(
      HTML("body {
           color: red;
      }")
    )
  ),
  ...
)

Esto solo es recomendable cuando la cantidad de elementos a customizar es corta, ya que esta sección puede crecer muy rápidamente y volver bastante confuso el código de nuestra interfaz.

En la sección head() también podemos incluir parámetros más específicos, como códigos escritos en javaScript para insertar animaciones/elementos interactivos, la descripción de la app, etc, por ejemplo:

fluidPage(
  tags$head(
    tags$title("Nombre de mi app"),
    tags$script(src = "codigo para animacion.js")
  ),
  ...
)

Debemos ser cuidadosos cuando añadamos códigos en JavaScript,

Típicamente, los tags que más comunmente usaremos en el desarrollo de la interfaz serán:

  1. Estructurales:
  • head(): Para añadir metadatos.
  • main(): Para priorizar contenido en una página.
  • header(): Para añadir encabezados.
  • footer(): Para añadir pies de página.
  • style(): Para aplicar formato CSS.
  • script(): Para incluir código en JavaScript.
  • link(): Para incluir referencias a otros códigos (principalmente de archivos CSS más largos).
  1. De formato:
  • p(): Para añadir un párrafo.
  • strong(): Para texto en negritas.
  • em(): Para texto en cursiva.
  • a(): Para hipervínculos.
  • img(): Para imágenes.
  • h1()-h6(): Para títulos/encabezados, siendo h1() el más grande y h6() el más pequeño.
  • br(): Para generar un espaciado entre elementos.
  • hr(): Para generar una línea horizontal divisora.
  • div(): Para una columna contenedora de otros elementos.
  • li(): Para listas ordenadas numéricamente (o alfabéticamente, usando el parámetro type="A"). Va escrito dentro del tag ol().
  1. Contenedores:
  • section(): Para definir secciones completas (vistas dentro de páginas). Útil cuando queremos aplicar estilos CSS de forma más general, afectando solo una parte específica de la app.
  • div(): Para columnas y subcolumnas. Suele estar dentro de section().
  • span(): Para alinear elementos.

Adicionalmente, también existen tags enfocados a la creación de elementos de tipo tabla, sin embargo, es poco probable que los utilicemos en shiny apps, ya que usualmente crearemos estos elementos a partir de recursos adicionales, como kable() de la librería knitr o reactable() de la librería con el mismo nombre.

9.1.7.2 CSS

Ya que tenemos la estructura simple de una app mediante el uso de HTML (usando tags o funciones predefinidas, como es el caso de fluidPage(), column(), etc), podemos comenzar a darle un estilo más agradable para el usuario final, utilizando código en lenguaje CSS (Cascading Style Sheets). Mediante CSS podemos modificar aspectos tales como el espaciado, tamaño de los márgenes, color y tamaño del texto (independientemente de que se trate de títulos o texto de párrafos), color del fondo de pantalla, entre muchos otros parámetros. Ahora, ¿cómo aplicamos el código CSS en una shiny app? Tenemos tres formas de hacer esto:

  1. Aplicándolo directamente en cada contenedor y elemento, por ejemplo:

    fluidPage(
  column(
    width = 1,
    style = "background-color:red;
             border: 10px;"
  )
)

    Nótese que, para poder aplicar un parámetro nuevo, tenemos que finalizar el parámetro anterior mediante un signo ;.

    Escribir el estilo del elemento HTML dentro del código de nuestra interfaz no siempre es la mejor opción, ya que tenderá a volver confuso el código conforme lo vayamos escalando.

  2. Aplicar el código CSS en la sección head(), por ejemplo:

    fluidPage(
    tags$head(
      tags$style(
        HTML(
          ".col-sm-5{
              background-color:red;
              border: 10px;
             }"
        )
      )
    ),
    column(
      width = 5,
      p("perro")
    )
  )

    Donde .col-sm-5 es la denominación dada al elemento column(width = 5), con la que podremos identificar a qué elemento vamos a aplicarle dicho cambio.

    En este caso estamos estractando todo el código CSS en una sola sección de la interfaz, lo que lo hace más fácil de interpretar pero aún nos limita a una corta escalabilidad, al tener que introducir demasiadas líneas de código dentro de la interfaz aún.

  3. Referenciar el código CSS únicamente en la interfaz, escribiéndolo en un archivo separado que guardaremos en la carpeta www, por ejemplo:

    fluidPage(
    tags$head(
      tags$link(rel = "stylesheet", 
                type = "text/css", 
                href = "estilo.css")
    ),
    column(
      width = 5
    )
  )

    Y, en el código CSS:

    .col-sm-5 {
  background-color:red;
  border: 10px;
}

Claramente, el uso de un archivo separado siempre será la mejor opción, ya que nos permitirá el añadir una cantidad considerable de elementos a nuestra interfaz, sin que esto se vuelva confuso conforme vayan aumentando.

Retomemos al ejemplo anterior, donde aplicamos un estilo a un elemento div con width=5. Este código tiene un problema: Este estilo se aplicará a todos los contenedores que tengan el parámetro width=5, ya que coinciden en el nombre de la clase. Es casi seguro que nos encontraremos con situaciones en las que solo necesitemos aplicar un cierto estilo a un elemento único, sin que los demás contenedores (u otros elementos) sean afectados por ello. Para poder aplicar estilos de una forma mucho más específica, simplemente tenemos que añadir un identificador (o ID) al elemento HTML que necesitemos trabajar y, del lado del código CSS, llamarlo con ese ID, por ejemplo:

column(
  id = "col_per",
  width = 5
)

Y, en el código CSS:

#col_per {
  background-color:red;
  border: 10px;
}

9.1.7.4 Plantillas (HTML templates)

Probablemente en más de una ocasión te has encontrado con un sitio web y has pensado algo como “wow, este sitio tiene un diseño increíble, quisiera adaptarlo a un proyecto mío”. Bueno, esto es posible en shiny apps. Así como podemos llamar código CSS en la sección head() de una app, podemos referenciar una plantilla completa e introducir elementos (inputs y outputs) diréctamente en un archivo de código HTML, ¿Cómo hacemos esto? Esto es un proceso bastante sencillo: Generalmente, cuando descargamos una plantilla HTML de internet, esta viene con múltiples archivos conectados entre si (códigos javaScript, customización CSS, imágenes, etc). Para adaptar la plantilla a nuestro proyecto, todos los archivos referentes a este template serán guardados con la misma estructura con la que los descargamos dentro de la carpeta www, que ya hemos mencionado con anterioridad. Una vez guardados, nos vamos a centrar en uno en particular llamado index.html, el cual contiene la estructura de la plantilla descargada, por lo que será el que vamos a referenciar en nuestra app. Suponiendo que ya tenemos el proyecto en la carpeta www, el código de nuestra app se verá de la siguiente manera:

shinyApp(
  ui = htmlTemplate(
    filename = "index.html"
  ),
  server = function(input, output, session){
    ...
  }
)

Ahora, del lado del código principal de la plantilla, supongamos la siguiente estructura en el documento index.html:

<!doctype html>
<html>
<head>
  <title>Ejemplo</title>
  <meta name="descripcion" content="Ejemplo">
  <meta name="ex" content="Ejemplo">
</head>
<body>
   Ejemplo de un texto simple.
</body>
</html>

Para poder añadir los elementos de nuestra app, debemos comenzar en la sección <head>, añadiendo el código headContent() de la siguiente manera (una plantilla real será mucho, muchísimo más grande, pero la estructura será muy parecida):

9.1.8 Dashboards con Shiny

Por supuesto, una de las mejores formas de presentar nuestros datos será dentro de un dashboard (tablero), ya que podremos condensar mucha información en varios páneles, de una forma entendible para el usuario final, pero ¿cómo lo podemos hacer en shiny? De manera estricta no necesitamos librerías adicionales para crear una visualización más al estilo de un dashboard, ya que en realidad lo hemos aplicado en los ejemplos que estuvimos trabajado anteriormente, al ser programas que dan un monitoreo y visualización sencilla de interpretar a ciertos datos, para que el usuario final pueda sacar cálculos y conclusiones de manera rápida (así como exportar la información), sin embargo, estos ejemplos han sido basdante básicos en cuanto a la interfaz usada, por lo que el objetivo de esta sección será el de entender cómo funcionan las librerías que están enteramente enfocadas a la creación de dashboards, cómo podemos adaptarlas a nuestros proyectos y, lógicamente, optimizarlas para su escalabilidad.

Para la creación de dashboards más especializados, las librerías más populares actualmente son shinydashboard, shinydashboardPlus, argonDash (en conjunto con argonR), tablerDash, bs4Dash, entre otras, paquetes que funcionan de manera muy similar, por lo que los siguientes ejemplos y desarrollos serán realizados con bs4Dash, haciendo uso del mismo set de datos walmart_data, que trabajamos en secciones anteriores.

Sí, los dashboards tienen como objetivo el centralizar los datos de un área del negocio o proyecto mediante resúmenes (gráficos, tablas, valores), concentrando una cierta cantidad de métricas en una sola vista. Sin embargo, eso no implica que debamos meter todos estos elementos en la misma página, ya que eso solo volvería confuso al tablero que estemos creando. Lo más óptimo siempre será dividir las vistas por tópicos. Además, existen distintos tipos de dashboards —estratégicos, tácticos y operativos— según el nivel de análisis y el público al que están dirigidos, aunque este tema no se abordará a fondo aquí. Por ejemplo: en un dashboard que esté destinado a mostrar los flujos de efectivo de una empresa, podemos dividir el tablero en: 1. Ingresos, 2. Costos y 3. Márgenes de ingresos, lo que evitará que el usuario que haga uso del dashboard se sature y se confunda con los números mostrados. 9.1.8.1 Gráficos, tablas y mapas basados en JavaScript

Ya que terminamos con los elementos generales del dashboard, vamos a continuar con la implementación del cuerpo, es decir, gráficos y tablas, pero en esta ocasión vamos a añadirle un toque extra. El uso de gráficos basados en la librería ggplot2 es genial, ya que podemos customizarlos de forma casi ilimitada, sin embargo, aún siguen siendo imágenes estáticas, por lo que obtener información en tiempo real de ellos se vuelve complicado, por ejemplo: En un gráfico donde no todas las barras se encuentren etiquetadas con el valor del eje y, será sumamente complicado inferir estos valores de forma exacta. Para poder obtener información adicional, así como realizar ciertas acciones dentro del mismo gráfico o tabla (sin utilizar inputs), podemos hacer uso de librerías basadas en código JavaScript, las cuales contienen Html Widgets, como se les denomina a este tipo de elementos visuales. En esa sección nos limitaremos al uso de tres librerías específicas para generar estos elementos interactivos: echarts4r2 (Echarts4r: Create Interactive Graphs with ECharts, n.d.) para gráficos, reactable3 para las tablas y mapboxer4 (Mapboxer: An r Interface to Mapbox GL JS, n.d.) para mapas.

En este caso, para acceder a los elementos reactivos (interpretados como inputs) tales como la barra seleccionada, o el punto en el que el puntero se encuentra posicionado, utilizaremos el id de identificación del elemento categorias_ventas como un prefijo, seguido del elemento que vamos a extraer, el cual puede ser:

  • clicked_data: Para datos seleccionados al hacer click en un elemento del gráfico. Se devolverá en un vector el valor de los ejes x y y.
  • clicked_row: Para el número de fila del dato seleccionado, correspondiente al data frame usado reactivo usado para crear ese objeto.
  • clicked_serie: Para el nombre de la serie seleccionada, si hay más de una.
  • mouseover_data: Para datos seleccionados al posicionar el mouse sobre un elemento del gráfico. Se devolverá en un vector el valor de los ejes x y y.
  • mouseover_row: Para el número de fila del dato en el que posicionamos el mouse, correspondiente al data frame usado reactivo usado para crear ese objeto.

Estos son los principales elementos que podemos extraer como objetos reactivos, aunque también tenemos disponibles otros elementos (brush, legend_change, legend_selected, clicked_data_value, mouseover_data_value y mouseover_serie) que usaremos con menos frecuencia.

9.1.8.1.4 Una pequeña reflexión

Estos tres ejemplos que analizamos son solamente tres librerías de cientos existentes con el mismo propósito, el de generar visualizaciones interactivas a partir de código JavaScript. Estos ejemplos únicamente tienen como propósito el mostrarte un funcionamiento general de los HTML Widgets, pero ¿por qué usarlos en lugar de simplemente imágenes estáticas que sean actualizadas mediante inputs? Realmente no es un requisito indispensable el usar este tipo de elementos en el tablero que estemos desarrollando, sin embargo, es preferible hacerlo en ciertos casos ya que la experiencia hacia el usuario final será mucho más agradable y le facilitará su uso. Cuando creamos un dashboard mediante shiny, debemos estar concientes en todo momento de que la información que vamos a presentar siempre debe ser concisa y fácil de entender para el usuario al que va a ir dirigida. Para nosotros como los desarrolladores del tablero, será sencillo el interpretar los datos que estamos expresando en este mismo, ¿Pero lo será para un usuario no técnico? Esa es la pregunta que siempre debemos tener en cuenta, ya que de esta forma nos será más sencillo el buscar soluciones visuales o interactivas que le permitan al usuario final el poder aprovechar la información del tablero de una mejor manera. Personalmente, me he encontrado en múltiples situaciones en las que hice un dashboard increíble, con prácticamente todas las métricas que mi compañero de trabajo necesitaba en ese momento, pero al presentárselo, su primer comentario se resumía en “¿Puedo exportar esto a Excel?”, ya que el dashboard era demasiado técnico, o no tenía el nivel de manipulación que mis colaboradores buscaban. Generalmente, cuando una persona no técnica utiliza un programa que cuenta con múltiples gráficos y mapas, casi siempre su primer acción será el dar click en algún punto del mapa, una barra de un gráfico o la columna de una tabla, esperando que algo pase con el resto de elementos del tablero, por lo que utilizar HTML Widgets nos será de gran utilidad, al poder satisfacer este tipo de situaciones, mejorando así la experiencia de uso.

Esta pequeña reflexión no es para que te desanimes respecto a la creación de tus tableros, si no para que entiendas y siempre tengas presente la razón por la que los hacemos, y como podemos crear un programa que sea de verdadera utilidad para la persona a la que va dirigido, sin que esta última piense en utilizar otra opción porque se le complicó utilizar el programa, lo que lo convertiría en un desarrollo sin ningún valor agregado.

9.1.8.2 Seguridad en mis dashboards

Este tema en particular es de vital importancia para los desarrollos que harás en el futuro. Probablemente tus primeros dashboards y/o programas serán elaborados con datos públicos o datos de muestra, ya que serán proyectos personales para tu portafolio, pero cuando trabajes en un dashboard para una empresa, por obvias razones deberás mantener la seguridad de este mismo como tu prioridad, de tal forma que puedas asegurarte que solamente las personas autorizadas tengan acceso a la información en este. Obviamente, no queremos que la información privada de una empresa u organización este expuesta al público o sea filtrada. En shiny tenemos algunas opciones para brindarle seguridad a nuestros tableros, siendo que en el presente libro nos vamos a enfocar en las Capas de Contraseña.

9.1.8.2.1 Capas de contraseñas

Comencemos por la opción más sencilla, siendo que actualmente ya existen librerías dedicadas a este aspecto, como lo es shinymanager (Shinymanager: Authentication Management for Shiny Applications, n.d.). Una capa de contraseñas no es más que un panel inicial que te solicita un usuario y una contraseña, tal como lo harías en cualquier página de uso cotidiano, como Facebook, Twitter, etc. Lo que hará esta capa de contraseñas es poner en pausa la renderización de todos los elementos de nuestro dashboard, y finalizar este proceso hasta que hayamos ingresado las claves correspondientes, dicho de otra manera, nuestro dashboard no va a iniciar ningún elemento hasta que se verifique que un usuario autorizado haya ingresado.

9.1.8.2.2 Control de accesos mediante bases de datos

Una forma de tener un mayor control respecto a los accesos que brindamos a nuestro dashboard es mediante un registro constante de los usuarios que acceden a este programa, pero ¿Cómo podemos obtener y guardar dicha información para poder explotarla después? Esto puede ser tan sencillo como enviar el registro de un acceso a una base de datos SQL, esto a partir de un evento en el servidor del dashboard. Comencemos por crear en la base de datos las tablas que contendrán la información de los usuarios y el registro de cada uno de ellos:

# Tabla de usuarios
CREATE TABLE dashboard_users(
  ID INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  user VARCHAR(30),
  nombre VARCHAR(30),
  password VARCHAR(30),
  rol VARCHAR(15),
  area VARCHAR(15),
  foto VARCHAR(30),
  status INT
);

# Tabla de registros (logs)
CREATE TABLE dashboard_logs (
  log_id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  log_timestamp TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES dashboard_users(ID)
)

Nosotros ya aplicamos dos reglas en la tabla de registros para que, tanto el log_id como la hora/fecha de acceso log_timestamp, se generen automáticamente, siendo que el log_id es un valor único incremental y la hora de registro es registrada como la hora actual en la que se crea la fila, por lo que bastará con mandar un query con el id del usuario que se registró en el servidor de nuestro programa, es decir:

server <- function(input, output, session){
  
  autenticacion <- secure_server(
    check_credentials = check_credentials(usuarios)

    if (!is.null(autenticacion$user)){
      dbSendQuery(conn, paste0("INSERT INTO dashboard_logs 
                                 (user_id) VALUES (",autenticacion$ID,")"))
    }
  })
  
}

Supongamos que la información de Márgenes es más sensible y, por ende, necesita un nivel de autorización mayor, ¿Cómo podemos evitar que otros usuarios accedan a este panel? Lógicamente, el primer paso será asignar los roles de acceso a los usuarios, lo que podemos hacer creando una columna adicional en nuestra base de datos con esta información:

ALTER TABLE dashboard_users
ADD access_level INT;

UPDATE dashboard_users 
SET access_level = 1
WHERE ID = 1;

Con esto, estamos suponiendo que nuestro usuario de ejemplo tiene un nivel de acceso igual a 1, pero para acceder al panel Márgenes se necesita un nivel de acceso igual a 2, por lo que el siguiente paso será dinamizar esta vista, de tal forma que solo se muestre a usuarios con los permisos requeridos. La forma más sencilla es plantear una función que englobe nuestra interfaz:

vista <- function(access_level,access_required,...){
  
  if (access_level < access_required){
    div(
      width = 12, align = "ceter",
      p("Sin permiso")
    )
  }else{
    tagList(...)
  }
  
}

Donde access_level será una variable que obtendremos de la tabla de usuarios, mientras que access_required será una variable que nosotros asignaremos, como el nivel mínimo requerido para acceder a dicho panel. Obviamente, si el usuario tiene un nivel de acceso menor al requerido, simplemente la función devolverá el mensaje “Sin permiso”. El argumento ... será un objeto que contenga la interfaz que vamos a dinamizar. Tonemos como ejemplo una interfaz en la que generemos un mapa que solo pueda ser visualizado por usuarios con un nivel de acceso igual o superior a 2, siendo el panel:

panel_condicional <-
  tagList(
    box(
      width = 12,
      align = "center",
      mapboxerOutput("mapa")
    )
  )

El cual será mostrado en la siguiente interfaz:

tabItems(
        tabItem(
          tabName = "ing_gen",
          uiOutput("panel_inicial")
        )
      )

Podemos guardar el objeto panel_condicional como una variable global, a la cual accederemos después mediante:

output$panel_inicial <- renderUI({
      
      vista(access_level = autenticacion$access_level, 
            access_required = 2, 
            panel_condicional)
      
    })

Obviamente, la tabla que contiene la información de acceso del perfil necesarioamente debe tener una columna llamada access_level, siendo esta la que estamos llamando en la función vista() para identificar el nivel de autorización del usuario que acceda, por ejemplo:

Por tanto, si el usuario Nombre_usuario ingresara al dashboard, sería capaz de visualizar los objetos dentro de este panel, debido a su nivel de autorización suficiente.

Ahora, manejar los accesos a los páneles bajo niveles de autorización puede no ser siempre la mejor opción, ya que si, es un método de seguridad confiable, sin embargo, al ser un método ambiguo respecto a los usuarios que tendrán acceso a cada panel, podemos dar pie a situaciones en las que un panel requiera un nivel de acceso elevado, pero un usuario con los niveles de acceso mínimos requeridos, pero perteneciente a otra área sea capaz de visualizar esta información. Para este tipo de situaciones podemos modificar la lógica de la función, agregando también otro tipo de condiciones más específicas, como el área a la que pertenece el usuario o el nombre del panel/páneles en específico al/los que tendrá acceso cada usuario, por ejemplo:

vista <- function(access_level,access_required,actual_department,
                  required_department,...){
  
  if (access_level < access_required 
      & required_department == actual_department){
    div(
      width = 12, align = "ceter",
      p("Sin permiso")
    )
  }else{
    tagList(...)
  }
  
}

Donde required_department será una constante en el servidor de nuestra aplicación que nos indicará a qué área pertenece el panel, mientras que actual_department será el valor obtenido a partir del usuario que esté ingresando, agarantizando que solo los destinatarios de la información puedan verla.

9.1.9 Shiny para (semi) automatizar tareas

Algo interesante de las Shiny Apps es el hecho de que no son específicamente dashboards, son aplicaciones web, lo que significa que la cantidad de cosas que podemos crear con ellas es ilimitada, estando únicamente sujetos a nuestra imaginación. Precisamente, en esta sección discutiremos como podemos implementar la lógica detrás de la semi automatización de procesos, así como ajustarla a cada caso particular.

Usualmente, solemos escribir códigos que siguen una cierta secuencia, por ejemplo, para generar un gráfico sencillo:

# Paso 1: leer los datos
data <- read.csv("ejemplo.csv")

# Paso 2: Transformar los datos
datos <- datos |> 
  group_by(x) |> 
  summarise(
    total = sum(y)
  )

# Paso 3: Generar el gráfico
grafico <- 
datos |> 
  ggplot(aes(x = x, y = total)) + 
  geom_col()

# Paso 4: Guardar el gráfico
ggsave("ruta_del_proyecto/grafico.pdf", plot = grafico)

Este código sigue una secuencia lógica que va a aplicar de la misma manera, cada vez que lo ejecutemos, sin embargo, no siempre tendremos abierta una sesión de R para correr el código, por lo que podemos contenerlo dentro de una Shiny App, ¿Cómo podemos hacer esto? Básicamente, podemos hacerlo de dos formas: Escribiendo directamente el código en el servidor de la app, o guardando el mismo en un script por separado y simplemente llamándolo desde el servidor. Ahora, obviamente ese código no se va a ejecutar solo por tenerlo en el servidor, debemos llamarlo de alguna manera, utilizando un evento, como es el caso de los botones. Veamos un ejemplo sencillo:

shinyApp(
  ui = fluidPage(
    actionButton("generar_grafico", "Generar gráfico")
  ),
  server = function(input, output, session){
    
    observeEvent(input$generar_grafico, {
      # Paso 1: leer los datos
      data <- read.csv("ejemplo.csv")
      
      # Paso 2: Transformar los datos
      datos <- datos |> 
        group_by(x) |> 
        summarise(
        total = sum(y))
      
      #Paso 3: Generar el gráfico
      grafico <- 
        datos |> 
        ggplot(aes(x = x, y = total)) + 
        geom_col()

      # Paso 4: Guardar el gráfico
      ggsave("ruta_del_proyecto/grafico.pdf", plot = grafico)
      
    })
    
  }
)

En este caso, estamos escribiendo directamente el código en el servidor, pero también podríamos guardarlo en un archivo separado llamado, por ejemplo, proceso.R, y llamarlo mediante el mismo evento:

shinyApp(
  ui = fluidPage(
    actionButton("generar_grafico", "Generar gráfico")
  ),
  server = function(input, output, session){
    
    observeEvent(input$generar_grafico, {
      
      source("proceso.R")
      
    })
    
  }
)

Ahora, esta muestra de código simplemente ejecuta un proceso estático, por lo que probablemente puedas pensar que esto no es útil en un uso práctico, ya que como programadores casi siempre tendremos nuestros códigos a la mano para ejecutarlos, lo que de hecho es cierto, por lo que ahora vamos a abordar un ejemplo mucho más completo que nos permita entender el potencial completo de los procesos que podemos ejecutar. Supongamos que los usuarios finales a quienes van dirigidos los análisis (stakeholders) están consumiendo constantemente un reporte, aunque de manera irregular, por lo que hacer uso de un JOB en la nube podría no ser la mejor solución en este caso, ya que no tenemos claro cada cuando tendremos que actualizar los datos de dicho reporte; Podemos aprovechar esta situación para generar un programa que le permita a los mismos usuarios finales el poder actualizar los datos que necesitan ellos mismos; Comencemos por establecer el proceso para extraer los datos que necesitamos, por ejemplo, si los datos vienen de una base de datos en SQL:

datos <- dbGetQuery("select ...

Supongamos ahora que estaremos enviando estos datos a un reporte en un documento colaborativo en Google Sheets, por lo que el siguiente paso lógico será:

write_sheet(data = datos, ss = "xxxxxxxxx", sheet = "Reporte")

Este proceso se puede volver aún más flexible, al dinamizar parámetros de la consulta de nuestro reporte. Vamos a convertir el código anterior en una función para poder añadir filtros más específicos en la consulta:

library(DBI)
library(googlesheets4)

shinyApp(

9.1.10 Buenas prácticas para estructurar una aplicación

Ahora que ya conocemos como crear aplicaciones con Shiny y todo lo que podemos hacer con ellas, vamos a terminar este capítulo aprendiendo las mejores prácticas para poder estructurar nuestras aplicaciones, haciéndolas escalables y sencillas de leer.

Algo común cuando comenzamos a crear nuestras primeras apps es que solemos mantener el código completo en un mismo archivo (casi siempre llamado app.R). Esto está bien cuando se trata de programas pequeños o prototipos simples. Sin embargo, a medida que nuestras aplicaciones crecen en complejidad (incorporando múltiples módulos, funciones, y elementos de UI) mantener todo el código en un solo archivo se vuelve inviable. Encontrar y modificar un componente entre cientos o miles de líneas de código se convierte en una tarea titánica y propensa a errores.

Como primera buena práctica, se recomienda dividir nuestro código en tres archivos principales:

  • global.R: Aquí se colocan todas las variables, funciones, conexiones a bases de datos, paquetes, y cualquier otro recurso que será compartido tanto por el servidor como por la interfaz de usuario. Este archivo se ejecuta una sola vez cuando se lanza la app, por lo que es ideal para tareas pesadas o costosas.

  • ui.R: Contiene exclusivamente el código relacionado con la interfaz gráfica del usuario. Aquí se definen los paneles, inputs, outputs, menús, estilos, temas, etc. Esta separación facilita que alguien con conocimiento en diseño o UX pueda trabajar directamente sobre este archivo sin alterar la lógica del servidor.

  • server.R: Aquí se define la lógica de la aplicación: los cálculos, transformaciones, llamados a la base de datos, generación de gráficos y respuestas a eventos. Este archivo contiene el server como función reactiva.

La estructura de la carpeta de nuestro proyecto se verá:

# /folder_de_mi_app/
# │
# ├── global.R
# ├── ui.R
# ├── server.R

Queda implícito el hecho de que es áltamente recomendable crear un proyecto que esté meramente destinado a nuestra aplicación, ya que tener guardados otros archivos en la misma ruta puede dar pie a confusiones al revisar/depurar el código, además de que será más dificil de mantener a largo plazo. Ahora, además de nuestros archivos principales, como ya mencioné en secciones anteriores, podemos incluir una carpeta www, en la cual guardaremos todos los objetos/archivos referentes a la customización de nuestra app, como pueden ser códigos CSS, imágenes, entre otros elementos. En el caso de incluir esta carpeta, la estructura de nuestra app será:

# /folder_de_mi_app/
# │
# ├── global.R           
# ├── ui.R               
# ├── server.R           
# │
# ├── www/               
# │   ├── styles.css     
# │   ├── logo.png

De esta forma, nuestro código es mucho más limpio y se vuelve sencillo el encontrar las secciones de la app que estamos buscando. Para referenciar estos archivos, de manera automática nuestro proyecto ya va a detectar que se encuentran dentro de la carpeta www, por lo que solo tendremos que nombrarlos sin ningún prefijo, por ejemplo:

tags$head(
  tags$link(rel = "stylesheet", 
            type = "text/css", 
            href = "styles.css")
)

Para llamar a un archivo CSS.

Si quisiéramos, podemos dividir las secciones de nuestra app aún más, para que incluso secciones pequeñas o más específicas se encuentren encapsuladas dentro de otros archivos separados, por ejemplo, imaginemos que tenemos un proceso de extracción de datos relativamente complejo y largo, por lo que escribirlo en el archivo global.R se vuelve poco práctico; En lugar de escribir ese código en la sección global, podemos escribirlo en un archivo separado llamado data.R y simplemente llamarlo desde el archivo global.R, por lo que el código de este último archivo se verá de la siguiente manera:

library(shiny)
...
...

source("data.R")

Volviéndose bastante sencillo de leer para otros usuarios, ya que ahora podemos identificar en qué archivo se encuentran cosas más específicas, como funciones, pipelines, e inclusive secciones del UI.

La contenerización de archivos en una shiny app no solo se limita a los aspectos que vimos anteriormente, si no también a secciones específicas de la app en si, como ventanas o páginas de la misma, cosa que podemos hacer separándola en módulos, tema que veremos a continuación.

9.1.10.1 Módulos

En una shiny app, los módulos permiten organizar el código en componentes reutilizables y escalables, lo que es ideal cuando la app empieza a crecer en complejidad y componentes. Básicamente, un módulo encapsula una parte de la interfaz y su lógica del servidor, para que puedas reutilizarla varias veces o mantener el código ordenado. El uso de módulos en shiny es especialmente útil cuando trabajamos colaboratibamente en un proyecto a gran escala, ya que cada usuario estará enfocado en desarrollar el código de su propio módulo, sin preocuparse por afectar la funcionalidad de los demás.

La estructura de los módulos es bastante sencilla, ya que simplemente vamos a englobar los componentes en dos funciones referentes al UI y el Server:

  • mod_funcion_UI(id): Para definir la parte de la interfaz.
  • mod_funcion_Server(id): Para definir la lógica del servidor.

Para ver cómo estructurar una aplicación mediante módulos, podemos retomar el ejemplo que vimos en la sección anterior. Para esta aplicación vamos a crear los siguientes archivos:

  • Global.R
  • UI.R
  • Server.R
  • Funciones.R
  • Modulo_panel_funciones.R

  tagList(
    dateInput(ns("fecha_inicial"), "Fecha inicial"),
    dateInput(ns("fecha_final"), "Fecha final"),

  )

# Servidor
mod_reporte_server <- function(id, reportes, conexion){
  moduleServer(id, function(input, output, session){
    
    updateSelectInput(session, "tipo_reporte", choices = names(reportes))
      
      withProgress(message = "Iniciando proceso",{
        observeEvent(input$generar_reporte,{
          req(input$tipo_reporte, input$fecha_inicial, input$fecha_final)
          
          funcion_reporte <- reportes[[input$tipo_reporte]]
          
          incProgress(0.5, message = "50% terminado")
          
          funcion_reporte(
            fecha_inicial = input$fecha_inicial,
            fecha_final = input$fecha_final,
            conexion = conexion
          )
          
          incProgress(0.5, message = "Proceso completado")
          
        })
      })
    
  })
}

Vamos ahora a analizar el código de los módulos. Del lado del UI es bastante sencillo, ya que simplemente creamos una función que llame a esa sección de la interfaz. Obviamente, como los elementos de HTML no se encuentran dentro de una función fluidPage(), tenemos que hacer uso de tagList(), para que puedan ser englobados y luego exportados a una sección de nuestra shiny app. Como puedes darte cuenta, además agregamos una línea de código adicional:

ns <- NS(id)

¿Qué significa? Este pequeño código hace referencia al NameSpace o espacio de nombre, y sirve para encapsular todo lo que se encuentre dentro de este módulo, evitando problemas de ambigüedad, como en situaciones en las que dos inputs de módulos diferentes tienen el mismo ID, lo que al mantenerlos separados mediante un NameSpace diferente, aunque ambos elementos tengan un mismo ID, este no se confundirá con el otro, permitiendo que nuestras apps sean mucho más escalables. Tanto del lado del UI como del Server tenemos como argumento la variable id, esta variable nos ayudará a asignarle el NameSpace a nuestro módulo para que sea identificable a partir de ella. Finalmente, puedes darte cuenta de que todos los inputs del módulo referente al UI se encuentran dentro de una función ns(), y esto es ocurre para que podamos mantenerlas única y exclusivamente dentro del NameSpace, siendo que todo lo que hagamos afuera de este espacio no lo va a afectar.

9.1.10.2 Otras funcionalidades de los módulos

Ya que trabajamos con la estructura general de los módulos, ahora vamos a exprimir sus capacidades un poco más. A pesar de que estos son independientes entre si, es posible que nos encontremos en situaciones en las que necesitemos exportar un cierto valor o input de un módulo a otro. El aislamiento entre estos objetos impide que utilicemos directamente el input$id_objeto de otro módulo, ya que esto rompería el funcionamiento de los módulos, por lo que la forma correcta de acceder a estos objetos es a través del uso de objetos reactivos. En lugar de acceder directamente al input, podemos hacer que un módulo devuelva un objeto reactive() o una reactiveValues() que luego se pasa como argumento a otro módulo, siendo este ya un objeto independiente. Veamos un ejemplo sencillo de esto, considerando la aplicación anterior con la que estuvimos trabajando: Imaginemos que ahora queremos crear un reporte adicional en otra sección de nuestra app, a partir de los parámetros del primer módulo, por lo que podemos hacer una modificación a este último para que exporte los valores reactivos en una lista:

mod_reporte_server <- function(id, reportes, conexion){
  moduleServer(id, function(input, output, session){
    
    updateSelectInput(session, "tipo_reporte", choices = names(reportes))
    
    valores <- reactiveValues()

    observeEvent(input$generar_reporte, {
      req(input$tipo_reporte, input$fecha_inicial, input$fecha_final)

      withProgress(message = "Iniciando proceso", {
        
        funcion_reporte <- reportes[[input$tipo_reporte]]
        
        incProgress(0.5, message = "50% terminado")

        funcion_reporte(
          fecha_inicial = input$fecha_inicial,
          fecha_final = input$fecha_final,
          conexion = conexion
        )

        incProgress(0.5, message = "Proceso completado")
        
        valores$fecha_inicial <- input$fecha_inicial
        valores$fecha_final <- input$fecha_final
        valores$tipo_reporte <- input$tipo_reporte
        valores$listo <- TRUE
      })
    })

    return(valores)
  })
}

De esta forma, con la ayuda del objeto valores <- reactiveValues(), podemos almacenar toda la información que necesitemos de forma reactiva. Ahora, vamos a crear el segundo módulo, el cual tomará estos parámetros y creará el reporte:

Para estos casos, observeEvent() solamente registra el primer disparador, lo que implica que cuando cambiemos de dirección (suponiendo que ya habíamos establecido una dirección previamente), los gráficos/tablas se van a actualizar sin que presionemos el boton por segunda vez, caso contrario a bindEvent(), ya que este último exige que hagamos click en el disparador en cada ocasión, lo que lo vuelve una mejor opción para este caso particular.

  1. Indicadores Clave de Rendimiento (KPIs)

Como encabezado, implementaremos tarjetas de información con métricas clave: ventas totales, volumen de órdenes, clientes únicos y ticket promedio. Aunque estos indicadores son fundamentales, por sí solos carecen de contexto temporal o comparativo.

  1. Apertura Geográfica y Filtros Dinámicos

Una tendencia general positiva podría estar sesgada por el éxito de una región específica, ocultando caídas en otras zonas. Para mitigar esto, integraremos un mapa interactivo que visualice las principales zonas de venta, complementado con filtros regionales. Así, el usuario podrá aislar y analizar la tendencia particular de cada localidad.

  1. Perfil Comercial y Eficiencia Operativa

Para responder de manera constante a preguntas críticas del negocio como “¿Cuál es la categoría más vendida?” o “¿Quiénes son nuestros principales compradores?”, añadiremos tablas de clasificación (top) de productos y clientes. Esta segmentación es vital para enfocar campañas de marketing y priorizar la distribución. Finalmente, incorporaremos una visualización del estatus de los pedidos (ciclo de vida del envío). Esto permitirá identificar cuellos de botella operativos y servirá como base para futuros análisis de optimización o maximización lineal en la cadena de entregas.

agregar_mapa <- function(df, granularidad) {
    group_cols <- if (granularidad == "ciudad")
      c("Direccion_str", "address_lng", "address_lat")
    else
      c("COUNTRY.y", "country_lng", "country_lat")
    
    df |>
      group_by(across(all_of(group_cols))) |>
      summarise(total = sum(SALES, na.rm = TRUE), .groups = "drop") |>
      mutate(radio = (total / sum(total)) * 100) |>
      setNames(c("Zone", "lng", "lat", "total", "radio")) |>
      filter(!is.na(lng), !is.na(lat))
}

    # columnas para Prophet
    prophet_data <- sales_ts |> 
      transmute(
        ds = as.Date(ORDERDATE_round),
        y = total
      )
    
    prophet_data
})
  
  
forecast_reactive <- reactive({
    
    # Modelo Prophet
    modelo_prophet <- prophet(
      modelo_reactive(),
      yearly.seasonality = TRUE,
      weekly.seasonality = FALSE,
      daily.seasonality = FALSE,
      seasonality.mode = "multiplicative",
      changepoint.prior.scale = 0.05,
      seasonality.prior.scale = 10
    )
    
    # Pronóstico a 12 meses
    future <- make_future_dataframe(
      modelo_prophet,
      periods = if (is.null(input$forecast_periods)) 12 
      else input$forecast_periods,
      freq = "month",
      include_history = TRUE
    )
    
    forecast_prophet <- predict(modelo_prophet, future)
    
    forecast_prophet
    
})

tabla_reactive <- reactive({
    
    historico <- modelo_reactive()
    forecast <- forecast_reactive()
    max_fecha <- max(historico$ds, na.rm = TRUE)
    
    x <- forecast |> 
      left_join(historico, by = "ds") |> 
      mutate(
        tipo = if_else(ds <= max_fecha, "Histórico", "Pronóstico"),
        fecha = format(ds, "%Y-%m"),
        `Ventas reales` = y,
        `Pronóstico` = yhat,
        `Límite inferior 95%` = yhat_lower,
        `Límite superior 95%` = yhat_upper,
        `Tendencia` = trend,
        `Tendencia inferior` = if ("trend_lower" %in% names(forecast)) 
          trend_lower else NA,
        `Tendencia superior` = if ("trend_upper" %in% names(forecast)) 
          trend_upper else NA,
        `Estacionalidad anual` = if ("yearly" %in% names(forecast)) 
          yearly else NA,
        `Términos aditivos` = if ("additive_terms" %in% names(forecast)) 
          additive_terms else NA,
        `Términos multiplicativos` = if ("multiplicative_terms" 
                                         %in% names(forecast)) 
          multiplicative_terms else NA
      ) |> 
      select(fecha, tipo, `Ventas reales`, `Pronóstico`,
             `Límite inferior 95%`, `Límite superior 95%`, `Tendencia`,
             `Tendencia inferior`, `Tendencia superior`,`Estacionalidad anual`,
             `Términos aditivos`, `Términos multiplicativos`) |> 
      arrange(fecha)
    
    x

output$download_forecast <- downloadHandler(
  filename = function() paste0("componentes_forecast_", Sys.Date(), ".xlsx"),

    n_rows <- nrow(x); n_cols <- ncol(x)

    cols_money <- c("Ventas reales", "Pronóstico", "Límite inferior 95%", 
                    "Límite superior 95%","Tendencia", "Tendencia inferior", 
                    "Tendencia superior", "Términos aditivos")

    money_cols <- which(names(x) %in% cols_money)
    pct_cols   <- which(names(x) %in% cols_pct)
    date_cols  <- which(names(x) == "fecha")

    wb <- createWorkbook(creator = "Shiny App")
    addWorksheet(wb, "Forecast", gridLines = FALSE)

    title_style <- createStyle(fontSize = 16, textDecoration = "bold", 
                               fontColour = color_header(), halign = "left")
    subtitle_style <- createStyle(fontSize = 10, fontColour = "#666666", 
                                  textDecoration = "italic")

                                fgFill = color_header(),
                                textDecoration = "bold", halign = "center", 
                                valign = "center",
                                border = "TopBottom", borderColour = "#D9D9D9")
    body_style  <- createStyle(fontSize = 11, fontColour = "#333333", 
                               border = "bottom", borderColour = "#EFEFEF")
    money_style <- createStyle(numFmt = '"$"#,##0', halign = "right")
    pct_style   <- createStyle(numFmt = "0.00%", halign = "right")
    date_style  <- createStyle(numFmt = "mmm yyyy", halign = "left")

    writeData(wb, "Forecast", "Componentes del pronóstico de ventas", 
              startRow = 1, startCol = 1, colNames = FALSE)
    mergeCells(wb, "Forecast", cols = 1:n_cols, rows = 1)
    addStyle(wb, "Forecast", title_style, rows = 1, 
             cols = 1:n_cols, gridExpand = TRUE)

    writeData(wb, "Forecast", paste("Archivo generado el", Sys.Date()), 
              startRow = 2, startCol = 1, colNames = FALSE)
    mergeCells(wb, "Forecast", cols = 1:n_cols, rows = 2)
    addStyle(wb, "Forecast", subtitle_style, rows = 2, 
             cols = 1:n_cols, gridExpand = TRUE)

    writeData(wb, "Forecast", x, startRow = 4, startCol = 1, 
              colNames = TRUE, withFilter = TRUE)
    addStyle(wb, "Forecast", header_style, rows = 4, 
             cols = 1:n_cols, gridExpand = TRUE)

    if (n_rows > 0) {
      body_rows <- 5:(n_rows + 4)

      addStyle(wb, "Forecast", body_style, rows = body_rows, 
               cols = 1:n_cols, gridExpand = TRUE)
      if (length(money_cols) > 0) addStyle(wb, "Forecast", 
                                  money_style, rows = body_rows, 
                                  cols = money_cols, gridExpand = TRUE, 
                                  stack = TRUE)
      if (length(pct_cols) > 0) addStyle(wb, "Forecast", 
                                  pct_style, rows = body_rows, 
                                  cols = pct_cols, gridExpand = TRUE,

      if (length(date_cols) > 0 && inherits(x$fecha, "Date"))

    }
    
    setColWidths(wb, "Forecast", cols = 1:n_cols, widths = "auto")
    setRowHeights(wb, "Forecast", rows = 1, heights = 28)
    freezePane(wb, "Forecast", firstActiveRow = 5, firstActiveCol = 1)

  }
